## Color

All tokens reference existing CSS variables from `index.css`. No new color values are introduced in this phase.

| Role | CSS Token | Light value | Dark value | Usage |
|------|-----------|-------------|------------|-------|
| Page background (60%) | `--background` | `oklch(1 0 0)` (white) | `oklch(0.145 0 0)` | Full-page login background |
| Card surface (30%) | `--card` | `oklch(1 0 0)` | `oklch(0.205 0 0)` | Login card itself |
| Card border | `--border` | `oklch(0.922 0 0)` | `oklch(1 0 0 / 10%)` | Card border, input borders |
| Accent (10%) | `--primary` | `oklch(0.205 0 0)` | `oklch(0.922 0 0)` | Submit button fill, focus ring |
| Destructive / Error | `--destructive` | `oklch(0.577 0.245 27.325)` | `oklch(0.704 0.191 22.216)` | Inline error text, error input border state |
| Muted text | `--muted-foreground` | `oklch(0.556 0 0)` | `oklch(0.708 0 0)` | Input placeholders |
| Focus ring | `--ring` | `oklch(0.708 0 0)` | `oklch(0.556 0 0)` | Input and button focus outlines |

Accent (`--primary`) reserved for:
1. Submit button fill (`bg-primary text-primary-foreground`)
2. Active focus ring on inputs and the submit button
3. Nothing else on the login page

Dark mode: login page is fully token-based — all elements use the above CSS variables, which automatically swap when the `.dark` class is applied to `<html>`. No hardcoded hex or oklch literals in component JSX.

## Component Specifications

### Login Card (`<LoginPage>`)

**Layout:**
- Full viewport: `min-h-screen bg-background flex items-center justify-center px-4`
- Card: `w-full max-w-sm` (384px) with shadcn `<Card>` primitive
- Card padding: `p-6` (24px all sides, using `<CardHeader>` + `<CardContent>`)
- Card has border and subtle shadow: shadcn Card default (`border border-border shadow-sm`)

**Logo slot:**
- Same logo-or-text logic as NavBar (Phase 5, D-05):
  - If `settings.logo_url` is non-null: `<img>` with `max-h-10 max-w-10 object-contain mx-auto`
  - If null: render nothing (the `<h1>` heading alone is sufficient)
- Logo sits above the `<h1>`, centered, with `mb-4` (16px) gap to heading
- Do NOT display app_name text alongside logo on the login card — login is identity-neutral

**Heading:**
- `<h1 className="text-2xl font-semibold text-center text-foreground">Sign in</h1>`
- Margin below heading: `mb-6` (24px) to the first form field

**Form (`<CardContent>`):**
- shadcn `<Form>` wrapping react-hook-form
- Field order: Email → Password → Submit button
- Gap between fields: `space-y-4` (16px)
- Each field: `<FormField>` → `<FormItem>` → `<FormLabel>` + `<FormControl>` wrapping `<Input>`
- Email input: `type="email"`, `placeholder="email@example.com"`, `autoComplete="email"`, `autoFocus`
- Password input: `type="password"`, `placeholder="Password"`, `autoComplete="current-password"`
- Submit button: full width `w-full`, `size="default"` (44px height), `variant="default"` (primary fill)

**Input states:**

| State | Visual |
|-------|--------|
| Empty/default | `border-input` (var --input), `bg-background` |
| Focus | `ring-2 ring-ring ring-offset-background` — shadcn default focus ring |
| Error | `border-destructive` + `aria-invalid="true"` on the input — shadcn FormMessage renders error below |
| Disabled (submitting) | `opacity-50 cursor-not-allowed` on all inputs and button — controlled by `disabled={isSubmitting}` |

**Submit button states:**

| State | Visual |
|-------|--------|
| Default | `bg-primary text-primary-foreground` |
| Hover | `bg-primary/90` (shadcn default) |
| Loading (submitting) | Label replaced by: `<Loader2 className="h-4 w-4 animate-spin mr-2" />` + "Signing in…" — button stays full-width, disabled |
| Disabled (submitting) | `disabled` attribute set; inherits `opacity-50` via shadcn |

**Inline error:**
- Rendered below the submit button as a `<p>` with `text-[13px] text-destructive text-center mt-2`
- Copy: `"Invalid email or password"` (locked — D-06 in 29-CONTEXT.md)
- Visible only when the Directus login call returns a non-2xx response
- Hidden (unmounted or `null`) when there is no error
- NOT a toast — no sonner call on login failure

### Full-Page Spinner (`<FullPageSpinner>`)

Used by `<AuthGate>` during `isLoading` (silent token refresh on app boot).

**Layout:** `min-h-screen bg-background flex items-center justify-center`

**Spinner element:** `<Loader2 className="h-8 w-8 animate-spin text-muted-foreground" />`
- 32px icon, using the existing lucide-react Loader2 icon (same pattern as submit button loading)
- `text-muted-foreground` — neutral, not `text-primary`, to avoid branded color on a loading gate

**No text label.** Internal tool, all users understand a spinner means "loading."

**Dark mode:** Fully token-based. `bg-background` and `text-muted-foreground` adapt automatically.

---

### NavBar Sign-Out Addition

The NavBar (Phase 5, `NavBar.tsx`) currently has: brand slot, SegmentedControl or back button, ThemeToggle, LanguageToggle, Upload icon, Settings gear.

**Sign-out placement:**
- Added to the right-side utility cluster (`ml-auto flex items-center gap-4`)
- Position: rightmost item, after the Settings gear
- Visual: `<button>` styled identically to the Upload and Settings icon buttons (same `inline-flex items-center justify-center rounded-md p-2 hover:bg-accent/10 transition-colors text-foreground` class pattern)
- Icon: `LogOut` from lucide-react (24px: `h-5 w-5`)
- `aria-label`: `"Sign out"` (English only — i18n is Phase 7's domain)
- No dropdown, no user email display, no avatar — plain icon button (deferred per 29-CONTEXT.md)

**Sign-out interaction sequence:**
1. Button press triggers `signOut()` from `useAuth()`
2. Button shows no loading state (sign-out is near-instant; failure still clears state)
3. After `signOut()` resolves (success or failure), router redirects to `/login`
4. No confirmation dialog — sign-out is not a destructive data action

**AdminOnly wrapping for Upload link:**
The Upload icon in NavBar (`<Link href="/upload">`) is wrapped with `<AdminOnly>`. Viewer users see the NavBar without the Upload icon entirely — not disabled, not grayed out, simply absent.

---

### `<AdminOnly>` Wrapper — Interaction Contract

Renders `null` for Viewer role. Renders children unchanged for Admin role.

**No visual chrome.** There is no disabled wrapper, no tooltip, no "locked" icon. The element simply does not exist in the DOM for Viewers.

**Affected surfaces (from D-08):**
1. NavBar Upload link icon
2. Upload page CTA buttons (dropzone submit, file select)
3. Personio sync trigger button
4. Settings page Save button + destructive controls
5. Delete icons on upload batch rows

**Viewer accessing `/upload` directly by URL:**
- The route remains mounted (no redirect)
- The full-width page content is wrapped in `<AdminOnly>`, replaced by a permission message
- Permission message: `<p className="text-muted-foreground text-sm text-center py-16">You don't have permission to access this page.</p>`
- Centered, uses existing token `text-muted-foreground`, 64px vertical padding

---

### 401 Redirect UX

No visual component. On 401 from `apiClient`, after silent refresh fails:
- Auth state clears (in-memory only)
- `setLocation('/login')` fires — wouter navigation
- No toast notification (D-09 in 29-CONTEXT.md: "No toast on 401 — route change communicates it")
- `/login` page renders fresh with no pre-filled state

## Mobile Responsive Breakpoints

The login page has one breakpoint: none needed for the card itself.

- `max-w-sm` (384px) card is narrower than any mobile viewport
- `px-4` on the outer container prevents edge-to-edge card on very narrow screens
- Card width: `w-full max-w-sm` — fills narrow viewports, caps at 384px on wider
- No responsive grid changes — login page is single-column only

The NavBar sign-out button addition has no mobile-specific behavior — it joins the existing right-side cluster. The NavBar itself has no responsive collapse in this project.
