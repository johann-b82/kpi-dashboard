This task has four sequential steps. Each step depends on the previous one succeeding.

**Step 1: Create .env from .env.example**

Copy `.env.example` to `.env`. The values in `.env.example` are suitable for local development (per D-06):
```bash
cp .env.example .env
```
The `.env` file is gitignored and will NOT be committed.

**Step 2: Generate initial Alembic migration**

Run Alembic autogenerate inside the Docker container to create the initial migration. This requires the database to be running, so start just the db service first:
```bash
docker compose up -d db
```
Wait for db to be healthy (the healthcheck is configured with pg_isready):
```bash
docker compose exec db pg_isready -U acm_user -d acm_kpi
```
Then run autogenerate from a temporary container using the backend image:
```bash
docker compose run --rm migrate alembic revision --autogenerate -m "initial schema"
```
This generates a migration file in `backend/alembic/versions/` with a random revision ID, containing:
- `op.create_table("upload_batches", ...)` with columns: id (Integer PK), filename (String 255), uploaded_at (DateTime timezone=True), row_count (Integer), status (String 20)
- `op.create_table("sales_records", ...)` with columns: id (Integer PK), upload_batch_id (Integer), col_a (String 255 nullable), col_b (String 255 nullable), col_c (String 255 nullable)

Verify the generated migration file contains both `create_table` calls. If the migration is empty, the model imports in `env.py` are not working -- check that `from app.models import Base` executes without error.

**Step 3: Bring up the full stack**

Stop the db-only setup and bring up everything:
```bash
docker compose down
docker compose up -d
```

Watch logs to confirm the startup sequence:
```bash
docker compose logs migrate
```
Expected: migrate container ran `alembic upgrade head` and exited 0. The log should show `INFO  [alembic.runtime.migration] Running upgrade  -> {revision}, initial schema`.

```bash
docker compose ps
```
Expected: `db` is running (healthy), `migrate` exited (0), `api` is running.

**Step 4: Verify end-to-end**

Test the health endpoint:
```bash
curl -f http://localhost:8000/health
```
Expected response: `{"status":"ok"}`

Verify tables exist in PostgreSQL:
```bash
docker compose exec db psql -U acm_user -d acm_kpi -c "\dt"
```
Expected: tables `alembic_version`, `upload_batches`, `sales_records` listed.

Verify column structure:
```bash
docker compose exec db psql -U acm_user -d acm_kpi -c "\d upload_batches"
```
Expected columns: id (integer), filename (character varying 255), uploaded_at (timestamp with time zone), row_count (integer), status (character varying 20).

```bash
docker compose exec db psql -U acm_user -d acm_kpi -c "\d sales_records"
```
Expected columns: id (integer), upload_batch_id (integer), col_a (character varying 255), col_b (character varying 255), col_c (character varying 255).

Verify data persistence:
```bash
docker compose down
docker compose up -d
curl -f http://localhost:8000/health
```
Expected: health check still returns `{"status":"ok"}` -- named volume preserved data.
