# Phase 27 Context ‚Äî FastAPI Directus Auth Dependency

**Goal:** FastAPI verifies Directus-issued JWTs (HS256 shared secret), resolves a `current_user` Pydantic model with `{id, email, role}`, and rejects unauthenticated/expired/malformed tokens with 401. Role gating is NOT in scope here (Phase 28).

**Requirements:** AUTH-01, AUTH-04, AUTH-05

## Locked decisions (from milestone + Phase 26)

1. JWT algorithm: **HS256** with shared `DIRECTUS_SECRET` (already in `.env` from Phase 26).
2. Directus admin role realized as `Administrator` (built-in), viewer as `Viewer` ‚Äî both with fixed UUIDs in `directus/bootstrap-roles.sh`.
3. No RLS; FastAPI is the single authz entry point.
4. Directus REST not exposed to browser ‚Äî frontend hits FastAPI only.

## Phase-27-specific decisions (locked via `/gsd:discuss-phase 27`)

### D-01 ‚Äî JWT library: `pyjwt`

Add `pyjwt==2.10.1` (or current latest 2.x) to `backend/requirements.txt`. Rejected `python-jose` (security advisories, less active) and `authlib` (overkill ‚Äî we only need HS256 verify, not a full OAuth client).

### D-02 ‚Äî Role matching: internal enum with UUID‚ÜíRole map

```python
# backend/app/security/roles.py
from enum import StrEnum

class Role(StrEnum):
    ADMIN = "admin"
    VIEWER = "viewer"
```

Mapping happens at JWT-decode time in `directus_auth.py`:

```python
_DIRECTUS_ROLE_UUID_MAP = {
    settings.DIRECTUS_ADMINISTRATOR_ROLE_UUID: Role.ADMIN,
    settings.DIRECTUS_VIEWER_ROLE_UUID: Role.VIEWER,
}
```

Downstream code uses `Role.ADMIN` / `Role.VIEWER` ‚Äî isolated from Directus's naming. If Directus renames `Administrator` ‚Üí `SuperAdmin`, only the `.env` UUID binding changes.

### D-03 ‚Äî `current_user` return type: Pydantic model

```python
# backend/app/schemas.py (or new backend/app/security/schemas.py)
class CurrentUser(BaseModel):
    id: UUID
    email: EmailStr
    role: Role
```

Matches existing `schemas.py` conventions (all request/response models are `BaseModel`). `Role` is `StrEnum` so JSON serialization is automatic.

### D-04 ‚Äî Unauthenticated endpoints: `/health`, `/docs`, `/openapi.json`

Every `/api/*` route requires a valid bearer. Exceptions:
- `GET /health` ‚Äî Docker healthcheck
- `GET /docs` + `GET /redoc` ‚Äî OpenAPI UI
- `GET /openapi.json` ‚Äî OpenAPI schema (required for `/docs` to work)

Planner must NOT add auth to these. The `/health` endpoint may need to be added in this phase if not yet present (scout step will confirm).

### D-05 ‚Äî Dep plumbing: router-level `dependencies=[Depends(current_user)]`

Every existing router (`data`, `hr_kpis`, `kpis`, `settings`, `sync`, `uploads`) gets `dependencies=[Depends(current_user)]` at the `APIRouter` constructor. Routes that need to *use* the user value still add `current_user: CurrentUser = Depends(current_user)` as a parameter ‚Äî cheap because FastAPI deduplicates.

Rationale: single place to flip auth on/off per router surface; no per-route boilerplate; accidental-omission-proof (can't ship a new endpoint in an authed router without auth).

### D-06 ‚Äî Directus role UUIDs via env vars

Two new settings in `backend/app/config.py` (Pydantic BaseSettings):

```python
DIRECTUS_SECRET: str
DIRECTUS_ADMINISTRATOR_ROLE_UUID: UUID
DIRECTUS_VIEWER_ROLE_UUID: UUID
```

Populated in `.env` and `.env.example`. Planner reads the Viewer UUID from `directus/bootstrap-roles.sh` (fixed `a2222222-bbbb-bbbb-bbbb-bbbbbbbbbbbb`). Administrator UUID must be looked up at plan-time against the running Directus (`GET /roles?filter[name][_eq]=Administrator`) ‚Äî it's generated by Directus on first boot and differs per install.

**Action for planner:** instruct executor to run a curl query against running Directus during plan execution, capture the UUID, document it in the plan as a data artifact, and write it to `.env` + `.env.example` (as a placeholder with generation instructions).

### D-07 ‚Äî Failure responses

Missing, expired, or malformed bearer ‚Üí 401 with body:

```json
{"detail": "invalid or missing authentication token"}
```

One message for all 401 cases (don't leak which check failed ‚Äî standard auth security practice). FastAPI's default `HTTPException(401)` body shape matches this.

### D-08 ‚Äî Test strategy: pyjwt-minted test tokens

Tests live in `backend/tests/test_directus_auth.py`. Test helper mints JWTs using the same `DIRECTUS_SECRET` and the same claim shape Directus emits. No live Directus, no mock library ‚Äî just `jwt.encode(...)`. Test cases:

1. Valid Admin token ‚Üí `current_user` resolves with `role == Role.ADMIN`
2. Valid Viewer token ‚Üí resolves with `role == Role.VIEWER`
3. Expired token ‚Üí 401
4. Wrong signature (signed with different secret) ‚Üí 401
5. Malformed bearer (non-JWT string) ‚Üí 401
6. Missing `Authorization` header ‚Üí 401
7. Unknown role UUID in claim ‚Üí 401 (defensive; shouldn't happen)
8. Valid token via `/health` ‚Üí 200 without auth (verifies D-04 exemption)

Phase 28 adds role-gating tests on top of this foundation.

## Sample Directus JWT claim shape (from Directus 11 docs)

```json
{
  "id": "2ff1a528-...",       // directus_users.id (UUID)
  "role": "aeccfcd7-...",      // directus_roles.id (UUID) ‚Äî this is what we map
  "app_access": true,
  "admin_access": true,
  "iat": 1712345678,
  "exp": 1712349278,
  "iss": "directus"
}
```

Note: `role` is a UUID, not the name. The email is NOT in the JWT. Planner must decide: either (a) do a one-time `GET /users/me` after login to fetch email, or (b) accept that `CurrentUser.email` is populated from the JWT-adjacent lookup.

**Recommendation for planner:** add email lookup via `GET /users/{user_id}` using a service-role token OR trust the frontend to pass email separately. Simplest: add a small async helper `fetch_user_email(user_id)` that hits Directus once per request with the user's own bearer (they have read access to their own record by default). Cache per-request via FastAPI `Depends` ‚Äî doesn't need Redis.

## Risk notes for planner

- **Directus JWT TTL is 15 min by default.** Tests must cover expired tokens with `iat - 20min`.
- **Email lookup adds one round-trip per request.** Mitigate with functools `lru_cache` keyed on user_id with short TTL (1-2 min). Not critical for 150 users but keep in mind.
- **Pydantic v2 strict mode** ‚Äî JWT claim UUIDs come as strings; `Field(validation_alias=...)` may be needed if claim names differ from field names.
- **`/health` endpoint**: scout confirms this does NOT exist currently (`main.py` has only `app.include_router(...)` calls). Planner should add a simple `GET /health ‚Üí {"status": "ok"}` as part of this phase ‚Äî required both for Docker healthcheck on api and for D-04 exemption test.
