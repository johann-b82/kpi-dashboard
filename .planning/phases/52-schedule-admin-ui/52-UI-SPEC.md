# Phase 52 — UI Design Contract

> Visual and interaction contract for the Schedule Admin UI. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Phase goal: admins create/edit/enable/disable/delete schedules via a 4th tab on `/signage`. DE/EN parity. Bilingual admin guide updated.
> Requirements: SGN-SCHED-UI-01..04.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn (already initialized — `frontend/components.json`) |
| Preset | `base-nova` (baseColor `neutral`, CSS variables, no prefix) |
| Component library | Radix (via shadcn primitives in `frontend/src/components/ui/`) |
| Icon library | lucide-react (per `components.json`) |
| Font | `Geist Variable` via `@fontsource-variable/geist` (project-wide `--font-sans`) |

**Existing shadcn components on disk** (reuse, do NOT re-install): `badge`, `button`, `calendar`, `card`, `checkbox`, `dialog`, `form`, `input`, `label`, `popover`, `segmented-control`, `separator`, `table`.

**Hard gates inherited from v1.16/v1.17 (all apply to this phase):**

1. DE/EN parity CI (`npm run check:i18n-parity`) — every new `signage.admin.schedules.*` key must exist in both `en.json` and `de.json`; DE copy uses informal "du" tone.
2. apiClient-only in admin frontend — schedule CRUD goes through `signageApi` / the shared `apiClient`; no raw `fetch()`.
3. No `dark:` Tailwind variants — semantic tokens only (`bg-background`, `bg-card`, `text-foreground`, `text-muted-foreground`, `border-border`, `bg-primary`, `text-destructive`, `bg-destructive`). `check:signage` grep guard enforces this.
4. `--workers 1` invariant (backend only; no UI impact).
5. Router-level admin gate (backend only).
6. No `import sqlite3` / `psycopg2` in frontend tree.
7. No sync `subprocess.run` in signage modules (backend only).

| Success / enabled indicator | `--color-success` | `#15803d` (project token) | "Enabled" badge on schedule list row (parity with existing DeviceStatusChip green) |

**Accent reserved for — explicit list (do not use elsewhere):**
1. Primary CTA button "New schedule" on the Schedules tab header
2. Primary CTA button "Save" / "Create" in the schedule editor dialog
3. Active tab indicator in the `SegmentedControl` when "Schedules" is selected
4. Focus ring on the currently focused form control (shadcn default `--ring` — derived)

**Destructive reserved for:**
1. Delete icon button in each row of the schedules table (`Trash2` lucide icon in `text-destructive`)
2. "Delete schedule" confirmation dialog primary button (destructive variant of shadcn `<Button>`)
3. Inline validation error text under a form field (e.g. "Start time must be before end time")
4. Inline 409 error surface when deleting a playlist blocked by active schedules (rendered on PlaylistsPage, not Schedules page — but uses same token)

Dark theme: Phase 52 MUST NOT add a `.dark` override; all values above are token-driven and the existing dark palette in `index.css` already handles them. `dark:` Tailwind variants are forbidden (hard gate 3, enforced by `npm run check:signage`).

### Interaction patterns (reuse existing)

- **List page layout:** identical structure to `PlaylistsPage.tsx` and `DevicesPage.tsx` — `<section>` → optional `<Button>` top-right → `<Table>` or empty-state `<section className="rounded-md border border-border bg-card p-12 text-center">`.
- **Create/edit:** dialog (not a separate route). Mirrors `PlaylistNewDialog.tsx` / `DeviceEditDialog.tsx` pattern.
- **Delete:** dialog confirmation, same shape as the media delete dialog (`MediaDeleteDialog.tsx`).
- **Toasts:** `sonner` — `toast.success(...)` / `toast.error(...)` — exactly as `DevicesPage.tsx:58-64`.
- **Loading:** TanStack Query `isLoading` gate; skeleton optional — existing tabs do not render skeletons, prefer consistency (no skeleton).
- **Empty state:** only rendered after first load resolves with zero rows (same guard as `DevicesPage.tsx:68`).
- **Refetch cadence:** no polling needed — schedule mutations fire SSE `schedule-changed` events (Phase 51 Plan 02). TanStack Query invalidates `signageKeys.schedules()` on mutation; the list is admin-only so there is no cross-user concurrency expectation. Do NOT add `refetchInterval`.

### Weekday bitmask adapter (D-05)

Bit positions: bit0=Monday ... bit6=Sunday. UI order MUST be Mo, Di, Mi, Do, Fr, Sa, So (matches German week start convention and is already implied by REQUIREMENTS.md SGN-SCHED-UI-02). Read: `boolean[7]` from `(mask >> i) & 1`. Write: `mask = checked.reduce((m, on, i) => on ? m | (1 << i) : m, 0)`.

### Time input format

Per SGN-SCHED-UI-02: two `HH:MM` text inputs (not separate hour/minute). Render format: `HH:MM` (24-hour, zero-padded). Wire format to backend: integer `HHMM` (e.g. `0700`, `1430`). Adapter: `hhmmFromString("07:00") → 700`, `hhmmToString(700) → "07:00"`. Use `<Input type="time">` for native picker + keyboard support (HTML5 time input returns `"HH:MM"` already).
