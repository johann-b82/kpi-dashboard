# Phase 47 — UI Design Contract: Player Bundle

> Visual and interaction contract for the standalone Pi-kiosk player bundle served at `/player/<device_token>`.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker before planning begins.

---

## Surface Inventory

The player bundle has exactly **three** rendered surfaces. Nothing else. No nav, no header, no footer, no toasts, no settings panel, no menu.

| Surface | Trigger | Layout | Chrome |
|---------|---------|--------|--------|
| **Pairing screen** | No token in URL AND no token in `localStorage` | Full-viewport, centered content | None — bare backdrop |
| **Playback canvas** | Valid token resolved (URL or localStorage) | Full-bleed, edge-to-edge | None — `<PlayerRenderer>` fills 100vw × 100vh |
| **Offline indicator chip** | `window.signageSidecarReady === true` AND sidecar reports `online=false` | Fixed positioning over playback | Single corner chip; auto-hides when online |

Loading / token-revoked transitions render the pairing screen as the fallback (D-2 § 47-CONTEXT). There is no separate spinner page — the brief gap is acceptable on a kiosk that boots once per day.

| Property | Value | Source |
|----------|-------|--------|

Grep guard for `dark:` in `frontend/src/player/**` must return zero (same rule as Phase 46).

## Spacing Scale

Standard 8-point scale. Player surfaces use a deliberately small subset.

| Token | Value | Tailwind Class | Usage in player |
|-------|-------|----------------|-----------------|
| xs | 4px | `gap-1` | Offline chip icon-to-text gap |
| sm | 8px | `p-2`, `gap-2` | Offline chip internal padding; corner offset clearance |
| md | 16px | `gap-4`, `p-4` | Pairing-screen hint paragraph margin from code |
| lg | 24px | `gap-6` | Pairing-screen hint to headline gap |
| xl | 32px | `m-8` | Offline chip distance from viewport edge |
| 2xl | 48px | `mb-12` | Pairing-screen vertical breathing room around the code block |
| 3xl | 64px | `gap-16` | Pairing-screen headline-to-code gap (visual hierarchy at distance) |

**Exceptions:** Playback surface uses ZERO spacing (full-bleed). `<PlayerRenderer>` fills `w-screen h-screen` with no padding, margin, or gap. Format handlers use `object-contain` so media is letterboxed against `bg-black` — this is the entire visual treatment of the playback canvas.

## Typography

The player has TWO typographic contexts: pairing chrome (read at 3–5m distance) and the offline chip (read at <1m if anyone walks up). Playback uses no typography of its own; whatever the media contains is what the audience sees.

| Role | Size | Weight | Line Height | Tailwind | Where |
|------|------|--------|-------------|----------|-------|
| Pairing headline | 60px | 600 (semibold) | 1.1 | `text-6xl font-semibold leading-tight` | "Pair this device" / "Verbinde dieses Gerät" |
| **Pairing code** | **256px** | **600 (semibold)** | **1.0** | **`text-[16rem] font-mono font-semibold tracking-[0.05em] leading-none`** | The `XXX-XXX` code — the entire reason this surface exists. ≥ 12rem per CONTEXT D-3; we go to 16rem (256px) because Pi 4 + 1080p screen at 3–5m readability testing pattern (Pitfall 13 + D-3 intent). |
| Pairing hint | 24px | 400 (regular) | 1.4 | `text-2xl font-normal` | One-line locale hint underneath the code |
| Offline chip text | 14px | 600 (semibold) | 1.4 | `text-sm font-semibold` | "Offline" / "Offline" label |

**Two weights total: 400 (regular) + 600 (semibold).** No `font-medium` (500) anywhere — same rule as Phase 46.

**Distance-readability rationale:** ROADMAP success criterion 2 + CONTEXT D-3 require the code to be readable from across a room. At 1920×1080 on a 50" panel, 256px ≈ 7cm character height — readable at 5m by typical vision. Headline at 60px serves as a smaller-but-still-clear orienting label. The pairing code is the only element that benefits from this scale; it is **not part of the general type scale and must not be reused.**

The rest of the kiosk has no other typography (playback is media-driven).

## Color

Player chrome uses a **fixed dark palette** rather than token classes. Justification: the kiosk has no theme switcher, no admin user, no light/dark contextual signal — it runs locked to dark forever, matching signage industry convention (dark backgrounds reduce ambient room glare from CRT/LCD bleed).

### Pairing screen palette

| Role | Value | Tailwind | Usage |
|------|-------|----------|-------|
| Dominant surface (60%) | neutral-950 | `bg-neutral-950` | Full-viewport pairing backdrop |
| Secondary (30%) | (none — pairing screen has no secondary surface; the dominant fills the viewport) | — | — |
| Accent / focus (10%) | neutral-50 | `text-neutral-50` | The pairing code itself, the headline, the hint — the only foreground content |
| Muted text | neutral-400 | `text-neutral-400` | (currently unused — pairing screen has no helper-tier text; reserved if a future "expires in N seconds" element is added per Pitfall 13) |

**60/30/10 honored as 60/0/40** (acceptable kiosk variance): the pairing screen has only two visual weights — a dark surface (60%+) and bright foreground content (~40%, dominated by the giant code). There is no tertiary chrome (no logo, no QR, no language toggle per D-3). The "10% accent" rule is reframed: the bright code IS the accent because it is the only thing the surface exists to show.

**Accent reserved for:** the pairing code, headline, and one-line hint on the pairing screen. Nothing else.

### Playback canvas palette

| Role | Value | Tailwind | Usage |
|------|-------|----------|-------|
| Letterbox / full-bleed surface | black | `bg-black` | Behind `object-contain` media; the canvas itself is just `<PlayerRenderer>` over a black ground |

Black (not neutral-950) for letterbox bars — matches video industry convention; pure black so projector/display blacks blend.

### Offline indicator chip palette

| Role | Value | Tailwind | Usage |
|------|-------|----------|-------|
| Chip background | neutral-900 / 80% opacity | `bg-neutral-900/80` | Subtle so it doesn't compete with playback content |
| Chip border | neutral-700 | `border border-neutral-700` | Defines the chip without harsh contrast |
| Chip text + icon | amber-400 | `text-amber-400` | Amber (not red) — "degraded but operating", reserved exclusively for offline status; matches v1.15 sensor amber convention. NOT destructive red. |
| Chip backdrop blur | blur-sm | `backdrop-blur-sm` | Survives over busy media without muddying it |

**Destructive color: not used.** The player has zero destructive interactions (no buttons at all on the kiosk). Token-revoked navigates to pairing screen without confirmation — this is correct: if the token is dead, the device cannot do anything else anyway.

## Interaction Contracts

### 1. Pairing screen (`/player/` with no token)

**Primary visual anchor: the `XXX-XXX` code block.** Everything else exists to support the code.

**Layout:** centered flex column, full viewport.

```
viewport (bg-neutral-950)
  └─ centered column (gap-16, items-center, justify-center)
       ├─ headline    "Pair this device" / "Verbinde dieses Gerät"   (text-6xl)
       ├─ code block  "ABC-123"                                       (text-[16rem] font-mono)
       └─ hint        "Enter this code in the admin panel under …"   (text-2xl text-neutral-400)
```

**Locale auto-detect:** `navigator.language` parsed at first paint. `de` or `de-*` → German; everything else → English. **No language toggle UI** (D-3). Detection happens once, before first render — no re-render flicker.

**Polling lifecycle:**
- On mount: `POST /api/signage/pair/request` → receive `{pairing_code, pairing_session_id, expires_in}`.
- Render the code immediately on response.
- Start `setInterval(3000)` polling `GET /api/signage/pair/status?pairing_session_id=…`.
- Polling can be implemented as TanStack Query with `refetchInterval: 3000` OR as a hand-rolled `setInterval` (CONTEXT § Claude's Discretion). Either is acceptable; planner picks per existing pattern in Phase 46-04 admin polling.
- On `{status: "claimed", device_token}`: write token to `localStorage.signage_device_token`, navigate to `/player/<device_token>`.
- On `expires_in` countdown reaching zero with no claim: re-issue `POST /pair/request`, update displayed code. (Phase 42 pairing TTL is 600s; Pitfall 13 mitigation.)

**No interactive elements.** No buttons, no inputs, no menus. The kiosk has no input devices during normal operation.

**Loading state (request in flight, no code yet):** render the headline + a small `text-neutral-400 text-2xl` placeholder em-dash (`—`) where the code will go. The viewport stays dark; no spinner. Acceptable because pairing-request latency is sub-second on the LAN.

**Error state (network failure on `/pair/request`):** retry every 5s silently; render headline + the small em-dash placeholder. **No error text shown to operator** — the kiosk is unattended; an operator who walks up sees the headline and knows the device hasn't paired yet, which is sufficient signal. (Field-debug overlay deferred per CONTEXT.)

### 2. Playback canvas (`/player/:token`)

**Primary visual anchor: the media itself.** Player chrome adds nothing.

**Layout:** `w-screen h-screen bg-black overflow-hidden` — `<PlayerRenderer>` fills it completely.

**Mount behavior:**
1. Resolve token (URL or localStorage per D-2). If absent → render `<PairingScreen>`.
2. Write token to `localStorage.signage_device_token` if it came from URL (D-2 persistence).
3. Mount `<PlaybackShell>`:
   - Initial `GET /api/signage/playlist` via apiClient + device JWT bearer.
   - Apply `durationDefaults` to items missing `duration_s` (D-6).
   - Pass items to `<PlayerRenderer>` (already auto-advances internally).
   - Open EventSource on `/api/signage/stream` (D-7).
   - Start watchdog timer (45s).
4. On every SSE event (including server's 15s `: ping` per Phase 45 D-02): reset watchdog. On `playlist-changed`: refetch playlist, swap items prop (renderer auto-resets currentIndex per 46-03 contract).
5. On watchdog fire OR EventSource `onerror`: close EventSource, start 30s polling loop (D-4). On successful poll: attempt SSE reconnect; if first event arrives within 5s, kill polling.
6. On any 401 from any API call: clear `localStorage.signage_device_token`, navigate to `/player/`. Pairing screen re-mounts.

**No heartbeat from JS** (D-8). Heartbeat is the Phase 48 sidecar's job. The bundle never POSTs `/heartbeat`.

**No user interaction.** Click events are not bound. Keyboard events are not bound. The cursor is hidden by the Pi-side `unclutter` daemon (Phase 48), not by CSS — but as a safety net, the playback root sets `cursor: none` on the body while playback is active.

**Format handler timing defaults** (filled before passing items to `<PlayerRenderer>`):

| Kind | Default `duration_s` if missing | Constant |
|------|--------------------------------|----------|
| image | 10 | `IMAGE_DEFAULT_DURATION_S` |
| video | (natural end via `onended`) | `VIDEO_DURATION_NATURAL` (sentinel — handler honors) |
| pdf | 6 × `pageCount` | `PDF_PER_PAGE_DURATION_S` |
| iframe (kind=url) | 30 | `IFRAME_DEFAULT_DURATION_S` |
| html | 30 | `HTML_DEFAULT_DURATION_S` |
| pptx | 8 × `slide_paths.length` | `PPTX_PER_SLIDE_DURATION_S` |

Constants live in `frontend/src/player/lib/durationDefaults.ts`. Single change-controlled file; one source of truth.

**Phase 47 disables the loop on `<VideoPlayer>`** (Phase 46-03 SUMMARY notes the admin-preview `loop` flag must come off in Phase 47). Mechanism: `<PlaybackShell>` advances on the video's `onended` event rather than on a timer; the renderer's video element is rendered without `loop`. (Implementation note for planner: this requires a small tweak to `<VideoPlayer>` to accept a `loop` prop, defaulting to `true` for backward-compat with admin preview, set to `false` by the player wrapper.)

### 3. Offline indicator chip

**Primary visual anchor: amber chip in bottom-right corner.**

**Position & sizing:**
- Fixed: `fixed bottom-8 right-8` (32px from each edge — comfortable margin from screen edge bezels)
- Size: `px-3 py-2 rounded-full` — pill shape; height ~36px
- Z-index: `z-50` — over playback content
- Backdrop: `bg-neutral-900/80 backdrop-blur-sm border border-neutral-700` — visible over busy media without dominating
- Content: `<WifiOff className="w-4 h-4" />` + label text — `flex items-center gap-1`
- Text: `text-sm font-semibold text-amber-400` — "Offline" (EN) / "Offline" (DE — same word)
- Aria: `role="status" aria-live="polite"` — accessibility hint for any screen-reader instance during dev/debug

**Visibility rules (D-1):**

| Sidecar status | Chip visible? |
|----------------|---------------|
| `window.signageSidecarReady === undefined` (no sidecar — dev server, browser preview, Phase 48 not yet shipped) | **Hidden** — the chip is a Phase 48-coordinated indicator; without the sidecar there is no authoritative offline signal |
| `window.signageSidecarReady === true` AND status reports `online=true` | **Hidden** |
| `window.signageSidecarReady === true` AND status reports `online=false` | **Visible** |
| `window.signageSidecarReady === true` AND status not yet read | **Hidden** (default to "online" until proven otherwise — avoids false positives on boot) |

**No transition animations.** The chip appears/disappears instantly. Fade-in is unnecessary chrome on a kiosk.

**No "online again" toast.** The chip just disappears when reconnected. Operators discover the recovery by glancing at the screen.

## Data Fetching Contract

All player queries use the `playerKeys` factory (extends `frontend/src/lib/queryKeys.ts`):

```ts
playerKeys.playlist()                  → ["player", "playlist"]
playerKeys.pairStatus(sessionId)       → ["player", "pair-status", sessionId]
```

| Query | Refetch strategy | Cache strategy |
|-------|------------------|----------------|
| `playerKeys.pairStatus(sessionId)` | `refetchInterval: 3000` while pairing screen mounted | `gcTime: 0` — flush on unmount |
| `playerKeys.playlist()` | `refetchInterval: false` normally; SSE event drives `invalidateQueries`. **30s `refetchInterval` activates ONLY when watchdog fires** (D-4 polling fallback). | Stale-while-revalidate via the Service Worker (D-1). In-memory: 5min `staleTime`, infinite `gcTime` (we never want to evict the last-known playlist). |

**apiClient with device JWT:** the player hooks attach `Authorization: Bearer <device_token>` to apiClient calls. This is the documented exception to the admin-only apiClient invariant — player uses apiClient with a *non-admin* bearer (CONTEXT 46 + ROADMAP cross-cutting hazard 2 explicit allowance: "Phase 47 player uses its own minimal fetch with device-token bearer, documented exception"). Implementation note: planner decides whether to (a) wrap apiClient with a small `playerApi` adapter that injects the device JWT, or (b) bypass apiClient and use `fetch` directly. **Recommendation: option (a)** — keeps the 401-handling path uniform with admin (which on player-side means "clear token + navigate to pairing").

**Service Worker runtime caching** (D-1, D-13):

```js
runtimeCaching: [{
  urlPattern: /\/api\/signage\/playlist/,
  handler: 'StaleWhileRevalidate',
  options: {
    cacheName: 'signage-playlist-v1',
    expiration: { maxEntries: 5, maxAgeSeconds: 86400 }
  }
}]
```

**Workbox precaches the bundle** (HTML/JS/CSS only). **Workbox does NOT precache media** (intentional — that's the Pi sidecar's job per D-1 + Pitfall 18).

## Copywriting Contract

All player copy under `signage.player.*` namespace. Both `en.json` and `de.json` must be updated simultaneously per the existing CI parity gate (Phase 46-04 script).

The player has **5 strings total**. Aggressively minimal by design.

### English copy

| Key | Copy |
|-----|------|
| `signage.player.pair.headline` | Pair this device |
| `signage.player.pair.hint` | Enter this code in the admin panel under Signage → Devices → Pair new device |
| `signage.player.pair.code_placeholder` | — |
| `signage.player.offline.label` | Offline |
| `signage.player.offline.aria_label` | Player is offline; cached content is playing |

### German copy (informal "du" tone)

| Key | Copy |
|-----|------|
| `signage.player.pair.headline` | Verbinde dieses Gerät |
| `signage.player.pair.hint` | Gib diesen Code im Admin-Panel unter Signage → Geräte → Neues Gerät koppeln ein |
| `signage.player.pair.code_placeholder` | — |
| `signage.player.offline.label` | Offline |
| `signage.player.offline.aria_label` | Wiedergabe läuft offline; Inhalte werden aus dem Cache gespielt |

**Empty state:** the player has no "empty playlist" UX surface visible to the audience. If the resolver returns zero items, `<PlayerRenderer>` already renders an em-dash placeholder (per 46-03 SUMMARY contract); on a kiosk this manifests as a near-black screen, which is the correct fail-soft behavior. Authoring an empty playlist is an admin-side mistake; surfacing operator-facing text on the audience-facing screen would be worse than the silent dash.

**Error state copywriting:** the player has no operator-facing error toasts (CONTEXT D-3 implicit, confirmed in this spec). All recoverable errors (SSE drop, 401, polling failure) trigger a state machine response — never a UI string. The single visible non-content state is the `Offline` chip.

**Destructive actions:** none. Token-revoked navigation to pairing screen is silent and correct.

**Locale fallback:** if `navigator.language` is not `de*` or `en*`, default to English. (Future locales out of scope per CONTEXT.)

## Accessibility Contract

The player runs in display-only kiosk mode with no input devices. ARIA contract is therefore narrowly scoped.

| Surface | Contract |
|---------|----------|
| Pairing screen | Headline is a `<h1>`. Code is rendered inside `<output aria-live="polite">` so any audio/screen-reader consumer (during dev, accessibility auditing, or testing) hears the code as it loads. Hint is a `<p>`. |
| Playback canvas | `<PlayerRenderer>` is `role="presentation"` — the audience-facing display is decorative from an a11y standpoint; media items contain no actionable structure. |
| Offline chip | `role="status" aria-live="polite"` so audits register the offline transition. |

Keyboard a11y: not required (no input devices). Focus management: not required (no interactive elements). Color contrast: pairing chrome (`text-neutral-50` on `bg-neutral-950`) ≥ 19:1 — far exceeds WCAG AAA. Offline chip (`text-amber-400` on `bg-neutral-900/80`) ≥ 8:1 against the chip's own background — passes AAA for normal text, AA for any rendering scenario.
