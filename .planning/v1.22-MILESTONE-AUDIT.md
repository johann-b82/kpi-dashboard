## Integration Findings

### INT-01 — toApiError adapter scoped to signage only (severity: medium, non-blocking)

The Phase 71 `toApiError()` helper exists at [frontend/src/lib/toApiError.ts](frontend/src/lib/toApiError.ts) and is correctly applied to all 30 Directus SDK call sites in `signageApi.ts`. However, the Phase 66 (auth) and Phase 67 (data) Directus call sites — which were migrated before the adapter was authored — are **not wrapped**:

- [frontend/src/auth/AuthContext.tsx:110-145](frontend/src/auth/AuthContext.tsx) — `directus.request(readMe(...))` and `directus.login(...)` re-throw raw `{errors:[{message,extensions}]}` plain-objects
- [frontend/src/auth/useCurrentUserProfile.ts](frontend/src/auth/useCurrentUserProfile.ts) — `readMe` not normalized
- [frontend/src/lib/api.ts:405-475](frontend/src/lib/api.ts) — `fetchSalesRecords` and `fetchEmployees` use `directus.request(readItems(...))` without `toApiError`

**Affects:** FE-01, FE-04, MIG-AUTH-01, MIG-DATA-01..03

**Why this is tech debt, not a blocker:** The adapter does normalize signage errors (the original FE-04 scope), and the contract-snapshot suite (71-03) tests transport shape on the happy path. The auth/data flows continue to function — they just bypass the locked error contract on failure paths.

### INT-02 — Cache namespace separation is one-shot eviction (severity: low, non-blocking)

The Phase 71 bootstrap purge ([frontend/src/bootstrap.ts](frontend/src/bootstrap.ts)) clears legacy `['signage', ...]` cache keys exactly once on cold start. But only DevicesPage was fully migrated to `['directus', 'signage_devices']` in Phase 70-04. The remaining signage admin surfaces (tags, schedules, playlists, playlist items, media, pair) still write into the legacy namespace via [frontend/src/lib/queryKeys.ts:67-78](frontend/src/lib/queryKeys.ts).

**Affects:** FE-02, FE-03

**Why this is tech debt, not a blocker:** The two namespaces don't collide (different prefixes), so the FE-02 invariant ("legacy stays independent, not reused") holds. The "namespace separation" is functionally complete for DevicesPage and a stylistic split for the rest.
