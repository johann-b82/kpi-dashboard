# MinIO for Outline Wiki Attachments — v1.11 Research

**Researched:** 2026-04-14
**Scope:** Single-node, single-drive MinIO as S3-compatible attachment backend for a self-hosted Outline wiki in a docker-compose stack.

---

## CRITICAL: MinIO Ecosystem Status (October 2025 – February 2026)

**Do not use `minio/minio` from Docker Hub.** The official MinIO project:

1. **October 23, 2025** — MinIO stopped publishing pre-compiled binaries and container images. The last Docker Hub image (`RELEASE.2025-09-07T16-13-09Z`) contains an unpatched CVE.
2. **December 3, 2025** — MinIO entered maintenance mode: no new features, no new PRs accepted, only critical security fixes considered case-by-case.
3. **February 13, 2026** — MinIO GitHub repository archived (read-only).

### Recommended replacement image

Use **`coollabsio/minio`** — a community project that automatically rebuilds MinIO from source and publishes to both Docker Hub and GHCR. Drop-in replacement; same CLI, same env vars, same ports.

| Source | Image ref |
|--------|-----------|
| Docker Hub | `coollabsio/minio:latest` |
| GHCR | `ghcr.io/coollabsio/minio:latest` |

**Alternative:** Chainguard's `cgr.dev/chainguard/minio:latest` (free tier, minimal CVE surface, continuously rebuilt from source). Slightly less documentation for the console workflow.

**Confidence:** HIGH — multiple independent sources (GitHub issue discussions, Minimus, Chainguard, Coolify project) corroborate the timeline.

## Minimum docker-compose Snippet

```yaml
services:
  minio:
    image: coollabsio/minio:latest
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: ${MINIO_ROOT_USER}
      MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD}   # min 8 chars
    volumes:
      - minio_data:/data
    ports:
      - "9000:9000"   # S3 API — only expose on loopback/internal if not using a reverse proxy
      - "9001:9001"   # Web console — close to public internet; admin only
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:9000/minio/health/live"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 15s
    restart: unless-stopped

  minio-init:
    image: coollabsio/minio:latest   # mc is bundled in the same image
    depends_on:
      minio:
        condition: service_healthy
    entrypoint: >
      /bin/sh -c "
        mc alias set local http://minio:9000 $${MINIO_ROOT_USER} $${MINIO_ROOT_PASSWORD} &&
        mc mb --ignore-existing local/${OUTLINE_BUCKET} &&
        mc anonymous set none local/${OUTLINE_BUCKET} &&
        mc admin user add local ${OUTLINE_S3_KEY} ${OUTLINE_S3_SECRET} &&
        mc admin policy attach local readwrite --user=${OUTLINE_S3_KEY} &&
        echo 'MinIO init complete.'
      "
    environment:
      MINIO_ROOT_USER: ${MINIO_ROOT_USER}
      MINIO_ROOT_PASSWORD: ${MINIO_ROOT_PASSWORD}
      OUTLINE_BUCKET: ${OUTLINE_BUCKET:-outline-attachments}
      OUTLINE_S3_KEY: ${OUTLINE_S3_KEY}
      OUTLINE_S3_SECRET: ${OUTLINE_S3_SECRET}
    restart: "no"

volumes:
  minio_data:
```

`.env` additions:
```dotenv
MINIO_ROOT_USER=minioadmin
MINIO_ROOT_PASSWORD=changeme123          # min 8 chars
OUTLINE_BUCKET=outline-attachments
OUTLINE_S3_KEY=outline-access-key
OUTLINE_S3_SECRET=outline-secret-key-min-8-chars
```

**Notes:**
- `--console-address ":9001"` is required — without it the console binds to a random port.
- `service_healthy` on the depends_on is required. MinIO takes a few seconds to accept API connections after the process starts.
- Port 9000 should stay internal (accessible only to Outline container on the compose network) unless you need browser-direct access to presigned URLs from outside. See "Outline gotchas" below.

## Bucket Bootstrap — Recommended Approach

**Use the `minio-init` one-shot service above (mc in entrypoint).** This is the standard approach and runs exactly once at startup, creating:

1. The bucket (`mc mb --ignore-existing` is idempotent — safe on restarts).
2. A scoped IAM user + readwrite policy restricted to that user.

### Why not the alternatives?

| Option | Verdict |
|--------|---------|
| Manual creation via MinIO console | Not reproducible; breaks on fresh deploys |
| Bitnami `MINIO_DEFAULT_BUCKETS` env var | Only works with `bitnami/minio` image, not the `coollabsio` image |
| `minio/mc` separate image | `mc` is bundled in the `coollabsio/minio` image; separate image adds complexity |
| Healthcheck + curl | MinIO has no built-in bucket-creation API on first boot |

## Credentials Model for Outline

**Use a scoped access key, not root credentials.** The `minio-init` service above creates a dedicated IAM user (`OUTLINE_S3_KEY` / `OUTLINE_S3_SECRET`) with the built-in `readwrite` policy. This user can read and write all buckets — which is fine for single-bucket single-service use.

If you want bucket-only scope, create a custom policy file and attach it:

```sh
# In the minio-init entrypoint, replace the policy attach step with:
cat > /tmp/outline-policy.json <<EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject","s3:PutObject","s3:DeleteObject","s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::${OUTLINE_BUCKET}",
      "arn:aws:s3:::${OUTLINE_BUCKET}/*"
    ]
  }]
}
EOF
mc admin policy create local outline-policy /tmp/outline-policy.json
mc admin policy attach local outline-policy --user=${OUTLINE_S3_KEY}
```

For a single internal team with one wiki, the built-in `readwrite` is acceptable and simpler.

**Do not use root credentials in Outline's config.** If root credentials are ever rotated or the MinIO service is restarted with new credentials, the scoped key persists independently.

## Outline Environment Variables

```dotenv
# File storage — choose S3 path
FILE_STORAGE=s3
AWS_ACCESS_KEY_ID=${OUTLINE_S3_KEY}
AWS_SECRET_ACCESS_KEY=${OUTLINE_S3_SECRET}
AWS_REGION=us-east-1                          # arbitrary; MinIO ignores region
AWS_S3_UPLOAD_BUCKET_NAME=${OUTLINE_BUCKET}
AWS_S3_UPLOAD_BUCKET_URL=http://minio:9000    # internal Docker network URL
AWS_S3_FORCE_PATH_STYLE=true                  # REQUIRED for MinIO
AWS_S3_ACL=private
FILE_STORAGE_UPLOAD_MAX_SIZE=26214400         # 25MB default; adjust as needed
```

**`AWS_S3_UPLOAD_BUCKET_URL` explanation:**
- Set this to the internal docker network address (`http://minio:9000`), not your external domain.
- Outline uses this URL to generate presigned URLs. If you expose MinIO externally via a reverse proxy, set this to the public URL instead so presigned links work from users' browsers.
- If MinIO is purely internal and users never access presigned URLs directly (Outline proxies attachment downloads), internal URL is fine.

## Outline-Specific MinIO Gotchas

### 1. Path-style addressing is mandatory (CRITICAL)
`AWS_S3_FORCE_PATH_STYLE=true` is required. MinIO defaults to path-style (`http://host:9000/bucket/key`). Without this flag, Outline sends virtual-hosted-style requests (`http://bucket.host:9000/key`) which MinIO rejects with 400/403.

### 2. Presigned URL domain mismatch
**Most common Outline + MinIO issue.** Outline generates presigned download URLs using `AWS_S3_UPLOAD_BUCKET_URL`. If that URL is `http://minio:9000` (internal docker host) but the browser tries to load the image, the browser gets a URL pointing to `minio:9000` which is not routable from outside Docker.

**Fix:** Set `AWS_S3_UPLOAD_BUCKET_URL` to the publicly accessible MinIO URL (e.g., `https://minio.yourdomain.internal:9000`). Port 9000 must be reachable from the user's browser.

### 3. CORS
MinIO does not automatically apply CORS rules. If Outline's frontend makes direct S3 PUT requests for uploads (it does, via presigned URLs), the browser will be blocked unless CORS is configured.

Add this to the `minio-init` entrypoint after bucket creation:

```sh
cat > /tmp/cors.json <<EOF
{
  "CORSRules": [{
    "AllowedOrigins": ["${OUTLINE_URL}"],
    "AllowedMethods": ["GET","PUT","POST","DELETE","HEAD"],
    "AllowedHeaders": ["*"],
    "MaxAgeSeconds": 3000
  }]
}
EOF
mc anonymous set-config local/${OUTLINE_BUCKET} /tmp/cors.json
```

Alternatively, set MinIO's global CORS Allow Origin via the console: Configuration → API → Cors Allow Origin → `https://your-outline-domain`.

### 4. `AWS_REGION` value
MinIO ignores region entirely but Outline requires this env var to be set. Use `us-east-1` as a placeholder — it works.

### 5. ACL private is correct
`AWS_S3_ACL=private` means objects are not publicly listable. Outline generates presigned URLs for access, which is the correct pattern for an internal wiki.

## Backup Story

MinIO's data directory (`/data` in the container, backed by the `minio_data` volume) is a standard filesystem. Objects are stored as regular files.

### Simplest approach: volume snapshot + optional mc mirror

**Option A — Volume snapshot (simplest, no extra tooling):**
```sh
# Stop or pause minio, then copy the volume
docker run --rm \
  -v minio_data:/source:ro \
  -v /backups/minio:/dest \
  alpine sh -c "cp -a /source/. /dest/"
```
MinIO SNSD (single-node single-drive) is documented as safe to back up this way — the filesystem layout is deterministic and self-contained.

**Option B — `mc mirror` (live backup, no downtime):**
```sh
# From host, using minio-init or a dedicated backup container:
mc alias set local http://localhost:9000 $MINIO_ROOT_USER $MINIO_ROOT_PASSWORD
mc mirror local/${OUTLINE_BUCKET} /backups/minio-mirror/
```
`mc mirror` is rsync-like: copies only changed objects. Does not preserve object version history, but Outline does not use versioning.

### Is data self-contained?

**No — MinIO and Postgres are interdependent for Outline.**

- MinIO stores the binary file data (images, attachments).
- Postgres stores the document content, attachment metadata, and the references (URLs) that link documents to MinIO objects.

You must back up both together. A MinIO backup without Postgres (or vice versa) produces an inconsistent restore. Schedule both backups to run at the same time or use a short maintenance window.

## Alternative: Local Filesystem Storage (Skip MinIO Entirely)

Outline v0.72+ supports `FILE_STORAGE=local`. This is the recommended choice for a single-node internal deployment that wants zero extra services.

```dotenv
FILE_STORAGE=local
FILE_STORAGE_LOCAL_ROOT_DIR=/var/lib/outline/data
FILE_STORAGE_UPLOAD_MAX_SIZE=26214400
```

```yaml
# In the outline service's volumes:
volumes:
  - outline_storage:/var/lib/outline/data
```

**Pros:**
- No extra container, no S3 config, no CORS issues, no presigned URL routing problems.
- Backup is a single volume copy.
- No image ecosystem problems.

**Cons:**
- Files are tied to the Outline container's volume — not independently addressable via S3 API.
- Cannot use CDN or external S3 tooling against the storage.
- If Outline is ever scaled to multiple replicas, local storage breaks (shared volume required).

**Recommendation for v1.11:** Unless you specifically need S3 compatibility or plan to integrate other tools against the object store, **local storage is the simpler and more reliable choice** for a single-node internal wiki. MinIO adds a container, a bootstrap init service, CORS config, and URL routing complexity that local storage avoids entirely.

Use MinIO only if:
- You want a web-browsable object store UI for attachment management.
- You plan to mirror attachments to an external S3 bucket.
- Another service in the stack also needs S3-compatible storage.

## Summary Recommendation

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Storage backend | `FILE_STORAGE=local` (preferred) or MinIO | Local is simpler; MinIO if S3 API needed |
| MinIO image | `coollabsio/minio:latest` | Official image abandoned Oct 2025 |
| Bucket bootstrap | `minio-init` one-shot service with `mc` | Idempotent, reproducible, no manual steps |
| Outline credentials | Scoped IAM user (not root) | Safer, independently rotatable |
| Path style | `AWS_S3_FORCE_PATH_STYLE=true` | Required — MinIO default |
| Backup | Volume snapshot + Postgres dump together | Must be consistent across both |
