## Phase Details

### Phase 38: Backend + Schema + Scheduler
**Goal**: The sensor data pipeline exists end-to-end in the backend — PostgreSQL schema is migrated, pysnmp polls seeded sensors on a scheduled cadence, admin-only `/api/sensors/*` routes return live data, and the community string is encrypted at rest and hidden from Directus. No UI.
**Depends on**: Phase 37 (existing AsyncIOScheduler singleton, Directus `require_admin` dep, Fernet key used for Personio credentials, `AppSettings` singleton pattern, Alembic head)
**Requirements**: SEN-DB-01, SEN-DB-02, SEN-DB-03, SEN-DB-04, SEN-DB-05, SEN-DB-06, SEN-DB-07, SEN-DB-08, SEN-BE-01, SEN-BE-02, SEN-BE-03, SEN-BE-04, SEN-BE-05, SEN-BE-06, SEN-BE-07, SEN-BE-08, SEN-BE-09, SEN-BE-10, SEN-BE-11, SEN-BE-12, SEN-BE-13, SEN-BE-14, SEN-SCH-01, SEN-SCH-02, SEN-SCH-03, SEN-SCH-04, SEN-SCH-05, SEN-SCH-06, SEN-OPS-01
**Success Criteria** (what must be TRUE):
  1. Running `docker compose exec api snmpget -v2c -c public 192.9.201.27 1.3.6.1.4.1.21796.4.9.3.1.5.2` from inside the `api` container returns a valid reading **before** any business logic is merged (GATING pre-flight for SEN-OPS-01)
  2. `alembic upgrade head` creates `sensors`, `sensor_readings`, `sensor_poll_log` and extends `app_settings` with sensor columns; `downgrade -1` cleanly removes them; `upgrade → downgrade → upgrade` round-trips on a fresh DB
  3. After migration, the seeded default sensor (Produktion / 192.9.201.27) exists and the scheduler ticks produce rows in `sensor_readings` every ~60 s with no duplicates under scheduled + manual poll collisions
  4. Every `/api/sensors/*` route returns 401 without a JWT and 403 with `{"detail": "admin role required"}` for a Viewer token; an admin can `GET /api/sensors`, `GET /api/sensors/{id}/readings`, `POST /api/sensors/poll-now`, `POST /api/sensors/snmp-probe`, `POST /api/sensors/snmp-walk`, and `GET /api/sensors/status`
  5. The Fernet-encrypted `community` field is ciphertext in Postgres (grep of `pg_dump` output returns zero plaintext community values), Directus Data Model UI does not list the `sensors` or `sensor_readings` tables, and Pydantic error responses never echo the community string
  6. A single shared `SnmpEngine` lives on `app.state.snmp_engine`; CI grep asserts no `import sqlite3`, no `import psycopg2`, and no `time.sleep` anywhere in `backend/app/services/snmp*`; router-dependency-audit test asserts every `/api/sensors/*` route chain contains `require_admin`
  7. `docker-compose.yml` runs uvicorn with `--workers 1` and carries the inline comment explaining why (prevents N-fold duplicate polls); `max_instances=1, coalesce=True, misfire_grace_time=30` on the sensor job; retention cleanup job prunes `sensor_readings` + `sensor_poll_log` older than 90 days
**Plans**: TBD
**Research flag**: no — STACK.md + ARCHITECTURE.md + PITFALLS.md + reference impl cover every surface

*Gating note: SEN-OPS-01 (Docker-to-SNMP-host reachability smoke test) runs as a pre-flight before any Alembic/service/router code lands. If the bridge cannot reach 192.9.201.x, the phase pauses and the host-mode fallback (Phase 40 runbook) is evaluated before proceeding.*

### Phase 39: Dashboard UI + Launcher Tile
**Goal**: An admin clicks the Sensors tile on `/home` and sees live sensor data — KPI cards per sensor with current values and threshold-aware badges, stacked time-series charts with reference lines, a working time-window selector, a Poll-now button that refreshes the view immediately, and delta/health chips. Viewers cannot see the tile at all.
**Depends on**: Phase 38 (all `/api/sensors/*` endpoints live, scheduler producing readings, Directus admin gate enforced)
**Requirements**: SEN-FE-01, SEN-FE-02, SEN-FE-03, SEN-FE-04, SEN-FE-05, SEN-FE-06, SEN-FE-07, SEN-FE-08, SEN-FE-09, SEN-FE-10, SEN-FE-11, SEN-FE-12, SEN-FE-13, SEN-LNCH-01, SEN-LNCH-02, SEN-LNCH-03, SEN-I18N-02
**Success Criteria** (what must be TRUE):
  1. An admin on `/home` sees a Sensors tile (Thermometer icon, bilingual `launcher.tile.sensors` label); a Viewer sees no such tile anywhere on the page; clicking the tile navigates to `/sensors`
  2. `/sensors` renders one KPI card per configured sensor with current temperature + humidity, threshold-aware badge colors (destructive token when out of min/max), per-card freshness footer, and empty/error state when the latest reading is null or carries `error_code`
  3. Two stacked Recharts `LineChart`s (°C and %) show one `Line` per sensor using the new `sensorPalette`, with dashed `ReferenceLine`s for global min/max thresholds; `connectNulls={false}` so gaps are visible
  4. A SegmentedControl with segments `1h · 6h · 24h · 7d · 30d` switches the time window via local state (or `SensorTimeWindowContext`) — `DateRangeContext` is **not** reused; `useSensorReadings` auto-refetches every 15 s when tab is visible, stops in background, refetches on window focus, and dedupes within 5 s
  5. Clicking Poll now blocks for up to 30 s, shows a success or failure toast, and invalidates all `sensorKeys` so cards and charts refresh immediately with the new row
  6. DIFF-01 delta badges render on each card (`+0.3 °C vs. 1 h`, `−2 % vs. 24 h`) via the `DeltaBadgeStack` pattern; DIFF-10 "OK seit Xh" / "Offline seit X min" health chip computes from `sensor_poll_log`
  7. Every surface uses Tailwind tokens only — no `dark:` variants, no hex literals outside the documented `sensorPalette` exception in `chartDefaults.ts`; all visible strings in the `sensors.*` dashboard-scope keys exist with full DE/EN parity in "du" tone; `launcher.tile.sensors` exists in both locales

### Phase 40: Admin Settings + Docs + Hardening
**Goal**: An admin can onboard a new sensor from the UI without touching SQL or YAML — discover OIDs via walk, validate via probe, save config, change the polling interval on the fly, and find complete bilingual operator documentation. The host-mode fallback is documented for operators whose Docker bridge cannot reach 192.9.201.x.
**Depends on**: Phase 39 (sensor dashboard live so admin can see their edits take effect), Phase 38 (all backend CRUD + probe + walk endpoints live)
**Requirements**: SEN-ADM-01, SEN-ADM-02, SEN-ADM-03, SEN-ADM-04, SEN-ADM-05, SEN-ADM-06, SEN-ADM-07, SEN-ADM-08, SEN-OPS-02, SEN-OPS-03, SEN-OPS-04, SEN-I18N-01
**Success Criteria** (what must be TRUE):
  1. From `/settings`, an admin can navigate to a dedicated `/settings/sensors` sub-page that applies the existing `SettingsDraft` + `UnsavedGuard` + `ActionBar` patterns; a Viewer attempting `/settings/sensors` is denied
  2. An admin can add, edit, and remove sensors via the CRUD form — each row exposes name, host, port, community (write-only `SecretStr`, never displayed), temperature_oid, temperature_scale, humidity_oid, humidity_scale, enabled; saving a CRUD edit persists to Postgres and the dashboard reflects the change on next refetch
  3. The collapsible "SNMP-Walk (OID-Finder)" section accepts host/port/community/base-OID, returns a results table, and allows click-to-assign an OID into a sensor row's temperature_oid or humidity_oid field; the per-row "Probe" button runs `POST /api/sensors/snmp-probe` on the uncommitted draft config and shows inline success/failure with live temp+humidity
  4. Changing the polling-interval input (integer seconds, 5–86400) and saving triggers `scheduler.reschedule_job(SENSOR_POLL_JOB_ID, ...)` in try/except; logs show the old interval, new interval, and computed `next_run_time`; the scheduled cadence changes without an API container restart
  5. Editing global thresholds (temperature_min/max, humidity_min/max — all optional NUMERIC) persists to `app_settings`, flows through `/api/sensors/readings`, and the dashboard reference lines + threshold badges update
  6. A bilingual admin-guide article `docs/admin/sensor-monitor.md` (EN) + `docs/admin/sensor-monitor.de.md` (DE) renders through the v1.13 docs pipeline and covers: onboarding a sensor (walk → probe → save), thresholds, polling interval, troubleshooting offline sensors, Docker bridge networking, the host-mode (`network_mode: host`) fallback runbook, the community-as-secret warning, and the never-use-default-public warning; both locale docs indexes list the new article
  7. All admin-scope `sensors.admin.*` i18n keys exist in DE and EN with full parity ("du" tone for German); dirty-guard prompt fires on navigate-away from `/settings/sensors` with unsaved changes using the existing `UnsavedChangesDialog`
**Plans**: 3 plans

- [ ] 40-02-PLAN.md — Interactive tooling: SNMP-Walk OID-Finder with click-to-assign, per-row Probe with inline result/toast, sensor-specific confirmation dialog, /settings link, walk/probe i18n, full admin onboarding human UAT

## Cross-Cutting Hazards (All Phases)

These are hard gates for every phase plan in v1.15. Phase plans must explicitly name how each is satisfied:

| Hazard | Rule | Enforcement |
|--------|------|-------------|
| **DE/EN i18n parity** | Every user-visible string must ship in both `en.json` and `de.json`; missing-key runs block the phase | Phase-plan checklist + pre-ship grep |
| **No direct `fetch()`** | All frontend network calls go through `apiClient<T>()` for bearer injection + 401-refresh-retry | ESLint / grep ban on `fetch(` under `frontend/src/` outside `apiClient.ts` |
| **No `dark:` variants** | Token-only Tailwind — colors come from CSS custom properties defined in `:root`/`.dark`; no hex literals outside documented `sensorPalette` exception | Grep ban on `dark:` and hex color regex in `frontend/src/` |
| **D-rule parity in phase plans** | Each phase plan enumerates the D-rules it inherits (e.g., D-10 root route lock) and asserts they are preserved | Plan-phase review gate |
| **`--workers 1` invariant in docker-compose.yml** | APScheduler is in-process with MemoryJobStore; N workers → N duplicate polls | Inline compose comment + router-dep-audit test indirectly enforces (duplicate sensors would fail unique constraint under repro) |
