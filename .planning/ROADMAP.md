### Phase 41: Signage Schema & Models
**Goal**: Alembic owns the signage schema end-to-end — 8 tables with the right indexes and FK semantics exist in Postgres, SQLAlchemy/Pydantic mirror them, and Directus introspects only the non-sensitive subset.
**Depends on**: Nothing (foundational; blocks every other v1.16 phase)
**Requirements**: SGN-DB-01, SGN-DB-02, SGN-DB-03, SGN-DB-04, SGN-DB-05, SGN-INF-02
**Success Criteria** (what must be TRUE):
  1. `alembic upgrade head` on a fresh DB creates `signage_media`, `signage_playlists`, `signage_playlist_items`, `signage_devices`, `signage_device_tags`, `signage_device_tag_map`, `signage_playlist_tag_map`, `signage_pairing_sessions` — `\dt signage_*` in psql lists exactly 8 tables.
  2. `\d signage_pairing_sessions` shows a partial-unique index on `code WHERE expires_at > now() AND claimed_at IS NULL`; attempting to delete a media row referenced by any `signage_playlist_items.media_id` fails with `FOREIGN KEY … RESTRICT`.
  3. `alembic upgrade head && alembic downgrade -1 && alembic upgrade head` round-trips cleanly with no residual tables, indexes, or constraints.
  4. After `docker compose up`, Directus Data Model UI exposes `signage_media`, `signage_playlists`, `signage_playlist_items`, `signage_device_tags` but does NOT list `signage_devices` or `signage_pairing_sessions`; `migrate` completes before `directus` starts (enforced via `depends_on.migrate.condition: service_completed_successfully`).
**Plans**: 5 plans
Plans:
- [x] 41-01-models-package-and-signage-models-PLAN.md — Split `models.py` into package; add 8 signage ORM classes (wave 1)
- [x] 41-02-schemas-package-and-signage-schemas-PLAN.md — Split `schemas.py` into package; add Pydantic v2 schemas for signage (wave 1)
- [x] 41-03-alembic-migration-PLAN.md — `v1_16_signage_schema.py` creating all 8 tables + partial-unique index + RESTRICT FK (wave 2)
- [x] 41-04-docker-compose-directus-exclusion-PLAN.md — `DB_EXCLUDE_TABLES` + `directus_uploads` RO mount on api (wave 1)
- [x] 41-05-round-trip-verification-PLAN.md — pytest round-trip test proving SGN-DB-01..05 (wave 3)
**Decisions resolved in planning**: Decision 2 — media storage — Directus owns user uploads via RO mount; backend owns PPTX-derived slides at `/app/media/slides/` (locked in CONTEXT D-01..D-03).

### Phase 42: Device Auth + Pairing Flow
**Goal**: A fresh Pi can display a code, an admin can claim it, and the resulting device token is the only way into `/api/signage/player/*` — pairing races, stolen tokens, and expired codes all fail safely.
**Depends on**: Phase 41
**Requirements**: SGN-BE-03, SGN-BE-04, SGN-SCH-02
**Success Criteria** (what must be TRUE):
  1. `curl -X POST /api/signage/pair/request` returns `{pairing_code: "XXX-XXX", pairing_session_id: <uuid>, expires_in: 600}`; `GET /pair/status?pairing_session_id=…` returns `pending` until an admin-authenticated `POST /pair/claim` with that code succeeds, after which the next poll returns `{status: "claimed", device_token}` exactly once.
  2. `curl /api/signage/player/playlist` with no Authorization returns 401; with a Viewer Directus JWT returns 401/403; with a valid device token returns 200.
  3. `curl -X POST /api/signage/pair/claim` with no admin token returns 401/403; unauthenticated `POST /pair/request` with >5 req/min from the same IP is rate-limited.
  4. APScheduler cleanup job (reusing 03:00 UTC slot) deletes `signage_pairing_sessions` rows with `expires_at < now() - 24h`; verified by seeding an expired row and asserting it's gone after the job runs.
  5. Admin "Revoke device" flips `signage_devices.revoked_at`; subsequent requests with the revoked token return 401.
**Plans:** 3/3 plans complete
Plans:
- [x] 42-01-device-auth-foundations-PLAN.md — Config + device_auth.py + rate_limit.py + pairing service (wave 1)
- [x] 42-02-signage-pair-router-PLAN.md — signage_pair router (/request, /status, /claim) + main.py wiring (wave 2)
- [x] 42-03-cleanup-cron-and-revoke-PLAN.md — 03:00 UTC cleanup cron + admin revoke endpoint + integration tests (wave 3)
**Decisions resolved in planning**: Decision 4 — device token format — scoped JWT HS256 24h TTL, NO rotation (re-issue on re-pair only); per-request revoked_at check is the kill path (locked in CONTEXT D-01..D-04).

### Phase 43: Media + Playlist + Device Admin API (polling)
**Goal**: Admins can CRUD signage content and devices via FastAPI; devices can resolve their current playlist and heartbeat over plain polling — polling alone is a functionally complete player loop before SSE lands.
**Depends on**: Phase 42
**Requirements**: SGN-BE-01, SGN-BE-02, SGN-BE-06, SGN-BE-09, SGN-BE-10, SGN-SCH-01
**Success Criteria** (what must be TRUE):
  1. `curl -X POST /api/signage/playlists` with admin JWT creates a playlist; same call with Viewer JWT returns 403 `{"detail": "admin role required"}`; no admin JWT returns 401 — enforced by `APIRouter(prefix="/api/signage", dependencies=[Depends(get_current_user), Depends(require_admin)])`.
  2. With a device tagged "lobby" and two enabled playlists both targeting "lobby" at different priorities, `GET /api/signage/player/playlist` returns the higher-priority playlist's items in order (LIMIT 1 via `priority DESC, updated_at DESC`).
  3. `POST /api/signage/player/heartbeat` with `current_item_id` updates `signage_devices.last_seen_at`; APScheduler sweeper (1-min cadence, `max_instances=1`, `coalesce=True`) flips devices with `last_seen_at > now() - 5 min` to `offline` status.
  4. Router dep-audit test passes: every admin signage route has `require_admin` in its dependant tree; every `/api/signage/player/*` route has `get_current_device`.
  5. CI grep guards pass: no `import sqlite3`, no `import psycopg2`, no `subprocess.run`/`subprocess.Popen` in `backend/app/` signage modules.

### Phase 44: PPTX Conversion Pipeline
**Goal**: A PPTX upload produces an ordered sequence of 1920×1080 PNG slides server-side without wedging the event loop, OOMing the container, or silently rendering with wrong fonts.
**Depends on**: Phase 43
**Requirements**: SGN-BE-07, SGN-BE-08, SGN-SCH-03, SGN-INF-01
**Success Criteria** (what must be TRUE):
  1. Uploading a valid 20-slide PPTX transitions `signage_media.conversion_status` through `pending → processing → done` within 60s on dev hardware; `signage_media.slide_paths` contains N image references in slide order; a known-corrupt PPTX transitions to `failed` within 60s without blocking a subsequent upload.
  2. Conversion runs via `asyncio.subprocess_exec("soffice", …)` wrapped in `asyncio.wait_for(60)` with `proc.kill()` on timeout, guarded by `asyncio.Semaphore(1)`, per-invocation tempdir + `-env:UserInstallation=file:///tmp/lo_<uuid>`; uploads >50MB are rejected at the API boundary.
  3. `docker compose build api` succeeds with `libreoffice-impress`, `libreoffice-core`, `poppler-utils`, Carlito, Caladea, Noto, and DejaVu fonts in the apt layer; `fc-list | grep -E 'Carlito|Caladea'` inside the container returns results.
  4. Scheduler startup resets any `signage_media` rows stuck in `processing > 5 min` (either to `failed` or re-queued per phase decision), verified by seeding a stuck row pre-restart.
**Plans**: 5 plans
Plans:
- [x] 44-01-dockerfile-libreoffice-fonts-PLAN.md — Dockerfile apt layer (LibreOffice + poppler + fonts) + /app/media/slides mkdir (wave 1)
- [x] 44-02-pptx-conversion-service-PLAN.md — signage_pptx service (Semaphore(1) + wait_for(60) + subprocess pipeline + state writes) + directus_uploads helper (wave 1)
- [ ] 44-03-upload-and-reconvert-endpoints-PLAN.md — POST /media/pptx multipart upload + POST /media/{id}/reconvert (wave 2, depends on 44-02)
- [x] 44-04-scheduler-stuck-row-reset-PLAN.md — _run_pptx_stuck_reset() one-shot startup hook in scheduler.py (wave 1)
- [ ] 44-05-integration-tests-PLAN.md — End-to-end integration tests (success/corrupt/stuck-reset) + fixtures + extended CI grep guards (wave 3)
**Decisions resolved in planning**: Decision 1 — PPTX worker location — LibreOffice runs in the `api` container via FastAPI BackgroundTasks (no dedicated pptx-worker); event-loop / OOM / font-fallback risks addressed by Semaphore(1) + wait_for(60) + per-invocation UserInstallation tempdir (locked in CONTEXT D-01). Isolated worker deferred to future phase per CONTEXT deferred list.

### Phase 45: SSE Broadcast
**Goal**: Admin saves propagate to connected players in under 2s via Server-Sent Events, while the `--workers 1` + APScheduler singleton invariants remain intact and polling remains a safety net.
**Depends on**: Phase 43
**Requirements**: SGN-BE-05, SGN-DIFF-01, SGN-INF-03
**Success Criteria** (what must be TRUE):
  1. With 5 concurrent SSE clients connected to `GET /api/signage/player/stream`, `curl /api/health` p95 latency remains <100ms; a server heartbeat `: ping` is emitted every 15s via `sse-starlette`.
  2. Admin `PUT /api/signage/playlists/{id}` that affects a device via tag overlap causes that device's SSE connection to receive a `playlist-changed` event within 1s; devices whose tags do not overlap receive nothing.
  3. `signage_broadcast.py` uses `asyncio.Queue(maxsize=32)` per device with `QueueFull` drop-on-overflow (30s polling covers the miss); module contains an inline comment block asserting the `--workers 1` invariant mirroring `docker-compose.yml` and `scheduler.py`.
  4. Killing the SSE connection from the client side cleans up the per-device queue in the generator's `finally` block — verified via a test that connects, disconnects, and asserts `_device_queues` has no stale entry for that device_id.

### Phase 47: Player Bundle
**Goal**: A Raspberry Pi opens `/player/<device_token>` in Chromium kiosk and reliably plays the current playlist — new uploads appear within 2s online, and the device keeps looping cached content when the network drops.
**Depends on**: Phase 42, Phase 43, Phase 45
**Requirements**: SGN-PLY-01, SGN-PLY-02, SGN-PLY-03, SGN-PLY-04, SGN-PLY-05, SGN-PLY-06, SGN-PLY-07, SGN-PLY-08, SGN-PLY-09, SGN-PLY-10, SGN-DIFF-03
**Success Criteria** (what must be TRUE):
  1. `npm run build` emits a separate `/player/index.html` bundle from a second Vite entry (`rollupOptions.input.player`); gzipped JS is <200KB; `GET /player/<device_token>` serves it with the token readable from the URL path and stored in `localStorage`.
  2. A Pi with no token lands on the pairing screen, calls `POST /api/signage/pair/request`, displays the code as `XXX-XXX`, polls `/pair/status` every 3s, and transitions into playback within 3s of an admin claim.
  3. Format handlers work: `<img>` with fade, `<video muted autoplay playsinline>`, pdf.js page-flip with 200ms crossfade (worker loaded via `?url` import, version-matched to `pdfjs-dist@5.6.205`), sandboxed iframe URL after HEAD pre-flight, nh3-sanitized HTML in sandboxed `<iframe srcdoc>`, PPTX as image sequence from `slide_paths`.

  5. With DevTools "Offline" enabled, the player keeps looping the last-cached playlist indefinitely; on reconnect, the next poll pulls updates. Service Worker (via `vite-plugin-pwa`) caches media Cache-API, stale-while-revalidate for playlist metadata.

## v1.16 Cross-Cutting Hazards (hard gates across all 8 phases)

These are non-negotiable invariants. Any phase plan that proposes to violate one of them is a blocker — escalate, do not work around.

1. **DE/EN i18n parity** — Every `signage.*` i18n key authored in `en.json` must exist in `de.json` (informal "du" tone). CI parity script (`scripts/check-i18n-parity.mjs`, v1.13 pattern) must be green before any admin-UI plan ships. Missing keys surface as literal keys on screen per v1.15 lesson. (Phase 46 + Phase 48 primary owners; every phase with UI touches checks.)
2. **No direct `fetch()` in admin frontend — apiClient only** — All signage admin requests go through the shared `apiClient<T>()` (v1.11 pattern) so bearer attach + 401-refresh-retry work. Grep for `fetch(` in `frontend/src/**/signage*` must return zero. (Phase 46 primary; Phase 47 player uses its own minimal fetch with device-token bearer, documented exception.)
3. **No `dark:` Tailwind variants** — Tailwind v4 token-only styling (`bg-background`, `text-foreground`, `bg-muted`, `border-border`). Dark mode works automatically via `:root` / `.dark` CSS variable blocks (v1.9 pattern). Grep for `dark:` in new signage components must return zero. (Phase 46 + Phase 47.)
4. **`--workers 1` invariant preserved** — `uvicorn --workers 1` stays literal in `docker-compose.yml`. SSE fanout via in-process `asyncio.Queue`, APScheduler singleton with `max_instances=1` + `coalesce=True`, and the PPTX semaphore all depend on it. Any plan proposing multi-worker is a blocker. Inline comments asserting the invariant land in `signage_broadcast.py` (Phase 45) mirroring existing blocks in `docker-compose.yml` and `scheduler.py`.
5. **Router-level admin gate via `APIRouter(dependencies=[…])`** — Admin signage routes composed as `APIRouter(prefix="/api/signage", dependencies=[Depends(get_current_user), Depends(require_admin)])` (v1.15 pattern). Per-endpoint gates are a smell. Dep-audit test (Phase 43, SGN-BE-09) walks the dependant tree and enforces this for every route.
6. **No `import sqlite3` / no `import psycopg2`** — Backend stays on async SQLAlchemy 2.0 + asyncpg. CI grep guard (Phase 43, SGN-BE-10) rejects either import anywhere under `backend/app/`.
7. **No sync `subprocess.run` in signage services** — PPTX conversion uses `asyncio.subprocess_exec` exclusively. Sync `subprocess.run`/`subprocess.Popen`/`subprocess.call` in `backend/app/` under any signage path is a blocker per PITFALLS Pitfall 2 (sync subprocess in async code blocks the event loop under `--workers 1`). CI grep guard enforces (Phase 43, SGN-BE-10; Phase 44 primary owner).
