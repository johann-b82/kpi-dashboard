**Researched:** 2026-04-14
**Confidence:** MEDIUM–HIGH

## 1. Scope Reality Check

Two significant findings changed the original scope:

**MinIO dropped — use `FILE_STORAGE=local`.**
The official MinIO Docker image was discontinued in October 2025 and the GitHub repository archived in February 2026. The `minio/minio` image on Docker Hub contains an unpatched CVE and receives no further updates. Community replacement images exist (`coollabsio/minio`) but add a container, a one-shot bootstrap init service, CORS configuration, and presigned URL routing complexity. Since Outline v0.72+ supports `FILE_STORAGE=local` (store attachments on the container filesystem via a named volume), that is the correct choice for a single-node internal deployment with 5–10 users. No S3 service needed. (Source: `v1.11-minio.md`, `v1.11-outline.md` §5)

**Dex RP-initiated logout is not available.**
Dex does not expose an `end_session_endpoint` in its OIDC discovery document. GitHub issue #1697 (open since 2020) requests it; it has not been merged as of April 2026. Cross-application simultaneous logout ("log out of KPI Light AND Outline in one click") is therefore not achievable via standard OIDC. The practical mitigation is local session logout in each app plus a short Dex token TTL (1–24h) to bound the window before stale SSO sessions expire. The FastAPI logout handler should check `end_session_endpoint` at runtime so it activates automatically if/when Dex ships it. (Source: `v1.11-kpi-oidc.md` §8, `v1.11-dex.md` §4)

## 3. Ordered Build Plan (Suggested Phases)

### Phase A — Dex IdP Setup

**Goal:** Deploy and verify Dex as the shared OIDC issuer. No app changes yet.

**Why first:** Everything downstream depends on a working, correctly-configured Dex. Issuer URL misconfigurations are the #1 integration failure point; catch them before adding app complexity.

**Deliverables:**
- `dex/config.yaml` committed to repo (with static passwords, two client registrations, SQLite storage)
- Dex service in `docker-compose.yml` with healthcheck + named volume
- Dex OIDC discovery endpoint reachable from both browser and internal containers
- At least one static user added, bcrypt command documented in runbook

**Complexity:** Low-medium. Config is dense but well-documented.

---

### Phase B — KPI Light OIDC Integration

**Goal:** Gate the existing KPI Light app behind Dex login; add auth-less dev mode toggle.

**Why second:** KPI Light is already running; adding auth to a known baseline is lower risk than adding a new app. Validates the Dex integration end-to-end with a smaller blast radius.

**Deliverables:**
- `authlib>=1.6.0` + `itsdangerous>=2.2.0` added to `requirements.txt`
- `SessionMiddleware` wired into `main.py`
- `backend/app/auth/` module: `client.py`, `router.py` (`/api/auth/login`, `/callback`, `/me`, `/logout`), `dependencies.py` (`get_current_user` with `DISABLE_AUTH` toggle)
- `AppUser` SQLAlchemy model + Alembic migration
- Frontend: `useCurrentUser` hook, `ProtectedRoute`, user display + logout in NavBar
- `DISABLE_AUTH=true` works in dev; production requires `DISABLE_AUTH=false` + Dex running

**Complexity:** Medium. Multiple files across backend + frontend; careful middleware ordering required.

---

### Phase C — Outline Wiki Deployment

**Goal:** Deploy Outline with its own Postgres + Redis, connected to Dex, using local file storage.

**Why third:** Dex is already verified. Outline's OIDC wiring requires a working issuer; having validated Dex with KPI Light first reduces debugging surface.

**Deliverables:**
- `outline-db`, `outline-redis`, `outline` services in `docker-compose.yml`
- `outline.env` (or equivalent) with all required vars including the internal/external OIDC URI split
- `outline_storage` volume mounted at `/var/lib/outline/data`
- Outline accessible via reverse proxy at `https://wiki.example.com`
- OIDC login working — user can log in to Outline with the same Dex credentials used for KPI Light

**Complexity:** Medium. The internal-vs-browser URL split for OIDC endpoints is the primary gotcha.

---

### Phase D — Wiki NavBar Integration

**Goal:** Add an icon/link from KPI Light's NavBar to the Outline wiki.

**Why fourth:** Depends on Outline being deployed and reachable at a stable URL.

**Deliverables:**
- NavBar icon (external link, opens in new tab) pointing to `VITE_WIKI_URL`
- `VITE_WIKI_URL` added to frontend env config

**Complexity:** Low. Minimal frontend change.

---

### Phase E — Seed Outline with KPI Light Docs

**Goal:** Author and publish the KPI Light documentation collection in Outline.

**Why fifth:** Outline must be live and stable before documents are seeded. Docs can be authored during Phase D but publishing requires a running wiki.

**Deliverables:** See Section 6 (Seed-Docs Inventory) for the full page list.
Tooling: Outline REST API (`/api/documents.create`) or GUI import; `getoutline-cli` (PyPI) for CI-based publishing if desired.

**Complexity:** Low-medium. Writing effort exceeds implementation effort.

## 5. Known Risks / Gotchas

**1. Issuer URL must be consistent across all three services (Dex, KPI Light, Outline).**
The `issuer` value in `dex/config.yaml` is baked into every ID token as the `iss` claim. KPI Light and Outline must configure their OIDC issuer to exactly match — same scheme, host, path, no trailing slash. The browser must also be able to resolve this URL. Never use Docker-internal hostnames (`dex`, `localhost`) as the issuer. (Source: `v1.11-dex.md` §5)

**2. Outline internal vs. browser URL split for OIDC endpoints.**
`OIDC_AUTH_URI` must be the public URL (browser redirects there). `OIDC_TOKEN_URI` and `OIDC_USERINFO_URI` are called by the Outline server container and should use internal Docker DNS (`http://dex:5556/...`) to avoid hairpin routing through the reverse proxy. Mixing these up is the #1 Outline OIDC misconfiguration. (Source: `v1.11-outline.md` §4)

**3. Dex static password changes require a container restart.**
There is no runtime user management. Adding/removing users means editing `config.yaml` and running `docker compose restart dex`. Deleted users' existing refresh tokens become invalid on next use, not immediately. For forced immediate revocation, wipe `dex.db`. Document the bcrypt hash generation command (`python3 -c "import bcrypt; print(...)"`) in the runbook so non-technical operators can add users. (Source: `v1.11-dex.md` §3)

**4. Session cookie 4KB limit.**
Starlette's `SessionMiddleware` encodes the entire session dict into a single signed cookie. Storing access tokens or refresh tokens there will hit the 4096-byte browser limit. Store only minimal claims: `{sub, email, name}`. (Source: `v1.11-kpi-oidc.md` §11)

**5. wouter `navigate()` vs `window.location.href` for OIDC redirects.**
`navigate("/api/auth/login")` is a client-side route change — FastAPI never sees it. Login redirects and logout form submissions MUST use `window.location.href = "/api/auth/login"` (full page navigation). (Source: `v1.11-kpi-oidc.md` §11)

**6. Dex logout gap — short TTL workaround.**
Since Dex has no `end_session_endpoint`, logging out of one app leaves the Dex session alive until the ID token expires. Mitigate by setting `expiry.idTokens: "1h"` in Dex config if cross-app logout coherence matters. For a low-stakes internal tool, the default 24h is acceptable and less annoying for users. (Source: `v1.11-kpi-oidc.md` §8, `v1.11-dex.md` §4)

**7. `https_only=True` breaks local HTTP dev.**
`SessionMiddleware(https_only=True)` causes the session cookie to be silently dropped on HTTP. Set `HTTPS_ONLY=false` in `.env` for local dev; enforce `true` in production. Wire it through `settings.https_only`. (Source: `v1.11-kpi-oidc.md` §11)

**8. Outline callback URL format.**
Outline's OIDC callback path is `/auth/oidc.callback` (dot, not slash). The `redirectURIs` in Dex config must be exactly `https://wiki.example.com/auth/oidc.callback`. (Source: `v1.11-dex.md` §8, `v1.11-outline.md` §4)

## 7. Open Questions for the Roadmap Phase

These require human decisions before or during phase planning:

1. **Reverse proxy choice.** Nginx or Traefik? Nginx is simpler config; Traefik offers automatic Let's Encrypt and dynamic service discovery. Which is already running (or preferred)?

2. **Domain names.** What are the public URLs for each service?
   - Dex: `https://auth.example.com` or `https://example.com/dex`?
   - Outline: `https://wiki.example.com` or `https://example.com/wiki`?
   - KPI Light: already set?

3. **Docs-as-code vs. wiki-only.** Should the seed docs also live as Markdown files in the git repo (version-controlled, auto-synced via `getoutline-cli` in CI)? Or is one-time manual seeding via GUI sufficient? This affects whether a CI pipeline step is worth setting up.

4. **Dex token TTL.** 24h (low annoyance, larger cross-app logout window) vs. 1h (more frequent re-auth, tighter logout coherence)? Depends on team preference.

5. **Outline version pin.** Research confirmed `0.86.0` as of 2026-04-14. Verify the current tag at `hub.docker.com/r/outlinewiki/outline/tags` before implementation — a `0.87.x` patch may be available. Do not jump to `1.0.0` without reviewing the migration notes.

6. **Dex `end_session_endpoint` verification.** Confirm Dex v2.43.0 does NOT expose `end_session_endpoint` by hitting `https://auth.example.com/.well-known/openid-configuration` on first deploy. If it does (Dex released a patch between research and deploy), update the logout flow accordingly.

### Gaps to Address

- **Reverse proxy config:** Not researched. The roadmapper should scope one phase to include Nginx/Traefik config if it is not already established in the environment.
- **TLS certificate provisioning:** Research assumed a working TLS-terminating reverse proxy. The mechanism (Let's Encrypt, self-signed, internal CA) was not specified and may need a sub-task.
- **Outline `1.0.0` vs `0.86.0`:** `v1.0.0` was released shortly after `0.86.0`. Research recommends `0.86.0` as the safer pin; review the `1.0.0` release notes before deploying if a more recent tag is desired.
