<objective>
Make the two-role configuration (`Admin`, `Viewer`) reproducible on any fresh stack. Commit `directus/snapshot.yml` as the source of truth, and add a `directus-snapshot` sidecar service to `docker-compose.yml` that applies the snapshot via Directus's `schema apply` CLI after the main `directus` service is healthy. The sidecar runs once per `docker compose up`, exits cleanly, and is a no-op if the snapshot is already applied.

Purpose: Close CFG-01 — a fresh machine's first `docker compose up` lands with both roles visible in the Directus admin UI, without operator intervention.

Output:
- New file `directus/snapshot.yml` with role definitions.
- `docker-compose.yml` gains a 6th service `directus-snapshot`.

Rationale for sidecar over `docker compose run`:
- A dedicated sidecar is declarative and runs automatically on every `docker compose up` — no separate command for operators to remember.
- `docker compose run --rm directus` would require a shell command on every bring-up (violates INFRA-01 "no manual steps between commands").
- Entrypoint wrapping the main `directus` service was rejected: mixing bootstrap logic into the long-running app container complicates debugging and slows cold boot.
</objective>

  <action>
Create `directus/snapshot.yml` defining the two roles required by CFG-01. Use fixed UUIDs so the snapshot is deterministic across machines (required — Directus resolves roles by ID, not name, so a name-only snapshot would create duplicates on re-apply).

Fixed UUIDs (generate once, commit forever — these are not secrets):
- Admin role: `a1111111-1111-1111-1111-111111111111`
- Viewer role: `a2222222-2222-2222-2222-222222222222`

File contents:

```yaml
version: 1
directus: 11.17.2
vendor: postgres
collections: []
fields: []
relations: []
roles:
  - id: a1111111-1111-1111-1111-111111111111
    name: Admin
    icon: verified
    description: Full access — manage users, roles, and all Directus system tables.
    admin_access: true
    app_access: true
    enforce_tfa: false
    ip_access: null
  - id: a2222222-2222-2222-2222-222222222222
    name: Viewer
    icon: visibility
    description: Read-only — can sign in and view own user record; no admin access.
    admin_access: false
    app_access: true
    enforce_tfa: false
    ip_access: null
permissions:
  # Viewer: read own user record only. No access to public.* app tables
  # (they are excluded from Directus entirely via DB_EXCLUDE_TABLES in Plan 01).
  - role: a2222222-2222-2222-2222-222222222222
    collection: directus_users
    action: read
    permissions:
      _and:
        - id:
            _eq: $CURRENT_USER
    validation: {}
    presets: null
    fields:
      - id
      - email
      - first_name
      - last_name
      - role
      - status
```

Notes on the schema (per D-02, CFG-01):
- `admin_access: true` on Admin grants unrestricted access to all Directus system tables (directus_users, directus_roles, directus_permissions, etc.) without needing explicit permission rows.
- Viewer has `admin_access: false` and ONE permission row: read-only on its own `directus_users` record (`id: $CURRENT_USER` filter).
- `collections`, `fields`, `relations` are intentionally empty — Directus discovers the `directus_*` system tables itself, and `public.*` app tables are excluded by `DB_EXCLUDE_TABLES` (Plan 01).
- `version: 1` is Directus's snapshot format version (not our app version).
- `directus: 11.17.2` matches the pinned image; if Directus rejects the snapshot on a version skew, regenerate via `docker compose run --rm directus npx directus schema snapshot --yes /directus/snapshot-mount/fresh.yml` and diff.

Snapshot drift risk (per CONTEXT risk notes): any role edits in the Directus admin UI will be OVERWRITTEN on the next `docker compose up`. Plan 03's checkpoint documents this as intended. Operators wanting a new role must edit `snapshot.yml` + commit; manual UI edits are ephemeral.

<task type="auto">
  <name>Task 2: Add `directus-snapshot` sidecar service to docker-compose.yml</name>
  <files>docker-compose.yml</files>
  <action>
Append a new `directus-snapshot` service to `services:` in `docker-compose.yml`, after the `directus` service added in Plan 01.

Service definition (insert verbatim):

```yaml
  directus-snapshot:
    image: directus/directus:11.17.2
    env_file: .env
    environment:
      KEY: ${DIRECTUS_KEY}
      SECRET: ${DIRECTUS_SECRET}
      DB_CLIENT: pg
      DB_HOST: db
      DB_PORT: 5432
      DB_DATABASE: ${POSTGRES_DB}
      DB_USER: ${POSTGRES_USER}
      DB_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - ./directus:/directus-snapshot:ro
    depends_on:
      directus:
        condition: service_healthy
    entrypoint: ["/bin/sh", "-c"]
    command:
      - "npx directus schema apply --yes /directus-snapshot/snapshot.yml"
    restart: "no"
```

Key design notes:
- Reuses the same `directus/directus:11.17.2` image — no new image to pull, guarantees CLI version parity with the main service.
- Shares DB env vars with `directus` so the CLI reads the same Postgres.
- Mounts `./directus` (host) read-only to `/directus-snapshot` (container). Using a separate mount path (not `/directus` — that's the image's working dir) avoids clobbering.
- `depends_on: directus: service_healthy` — snapshot apply only runs AFTER Directus has booted, migrated its own schema, and passed its `/server/health` check. This is CRITICAL: `schema apply` needs `directus_roles` / `directus_permissions` tables to exist.
- `entrypoint: ["/bin/sh", "-c"]` overrides the image's default (which starts the Directus server). We want one-shot CLI.
- `restart: "no"` — the sidecar exits 0 after apply; without this, Compose's default behavior on some setups would restart it.
- Idempotence: `schema apply --yes` diffs the snapshot against live state and applies only deltas. Re-running with an unchanged snapshot is a near-instant no-op. This is why running on every `docker compose up` is safe.

Do NOT add `directus-snapshot` to `api` or `frontend`'s `depends_on` — those services don't need the snapshot to boot. The snapshot is for operator-facing role state, not runtime.
  </action>
