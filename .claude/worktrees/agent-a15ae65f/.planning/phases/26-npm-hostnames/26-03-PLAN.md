  <action>
Create `docs/setup.md` (new file — `docs/` directory does not yet exist, create it). Write the runbook in a form a new developer can execute top-to-bottom. Use these sections:

1. **Prerequisites** — Docker Engine + Docker Compose v2 (`docker compose`, not `docker-compose`), mkcert (link to https://github.com/FiloSottile/mkcert with install commands for macOS `brew install mkcert` and Debian/Ubuntu `apt install libnss3-tools` + mkcert binary download), a modern browser.

2. **One-time machine setup**
   - Run `mkcert -install` once per developer machine (explain: installs the mkcert CA into the system + browser trust stores, which is why `https://kpi.internal` shows a green padlock with no click-through — D-02).
   - Edit `/etc/hosts` (or `C:\Windows\System32\drivers\etc\hosts` on Windows) and add the exact line:
     ```
     127.0.0.1 kpi.internal wiki.internal auth.internal
     ```
     (This is INF-03. Document this line verbatim, in a code block, so copy-paste is safe.)

3. **Per-checkout setup**
   - Clone the repo.
   - Copy `.env.example` to `.env` and fill in any secrets (refer the reader to the existing POSTGRES_* vars; note that NPM needs no env vars per D-09).
   - From repo root: `./scripts/generate-certs.sh` — explain that this produces `./certs/internal.{crt,key}` and that the files are gitignored.
   - `docker compose up --build -d` — the full stack boots. Expected state: `db`, `api`, `frontend`, `npm` all `healthy`; `migrate` exited 0.

4. **First-time NPM admin bootstrap** — mirrors the 26-02 checkpoint steps (D-09, D-10). Steps:
   a. Open `http://localhost:81` in a browser.
   b. Log in with `admin@example.com` / `changeme`. You will be prompted to change both the email and password immediately. Do so and store the new password securely.
   c. Go to **SSL Certificates > Add SSL Certificate > Custom** and upload `./certs/internal.crt` (Certificate) + `./certs/internal.key` (Key). Name it `internal-wildcard`.
   d. Go to **Hosts > Proxy Hosts > Add Proxy Host** and create the three hosts. For each: use the `internal-wildcard` cert on the SSL tab with "Force SSL" + "HTTP/2 Support" enabled. Specifics:
      - **kpi.internal** — Scheme `http`, Forward Hostname `frontend`, Forward Port `5173`. Enable **Websockets Support**. On the **Advanced** tab, paste:
        ```
        location /api {
            proxy_pass http://api:8000;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
        ```
        (This is the critical INF-02 wiring — the /api sub-location bypasses Vite and hits FastAPI directly.)
      - **wiki.internal** — Placeholder until Phase 29. Forward to the NPM default page or configure as "Offline" with a custom 503 message. Exact choice and walkthrough per 26-02-SUMMARY.md.
      - **auth.internal** — Placeholder until Phase 27. Same treatment as wiki.internal.

5. **Verification** — browser tests matching the 26-02 checkpoint:
   - `https://kpi.internal` → KPI Light dashboard loads, green padlock, HMR works on file edit.
   - `https://kpi.internal/api/health` → 200 JSON response.
   - `https://wiki.internal` / `https://auth.internal` → reachable (placeholder or 503).

6. **Troubleshooting** — table of common failures:
   - Padlock warning / cert error → `mkcert -install` was not run, or browser hadn't been restarted since install.
   - HMR WebSocket failures in DevTools console → NPM proxy host for kpi.internal does not have "Websockets Support" enabled.
   - `502 Bad Gateway` on kpi.internal → `frontend` or `api` container not healthy; check `docker compose ps`.
   - `/api/*` returns HTML instead of JSON → the Advanced `location /api` block on the NPM proxy host is missing or wrong.
   - "Blocked request. This host ("kpi.internal") is not allowed." → `allowedHosts` in vite.config.ts is missing kpi.internal (regression from 26-02).

7. **Future (v2 — AUTH2-03)** — one short paragraph: when a real DNS domain is available, swap the mkcert custom cert for a Let's Encrypt DNS-01 cert via NPM's built-in LE flow. The `npm_letsencrypt` volume created in Phase 26 is already mounted in anticipation. Self-signed remains the default for single-VM v1.11 (D-04).

Keep the tone matter-of-fact and pragmatic. No marketing language. Use fenced code blocks for every command. Every filesystem path must be absolute from repo root or clearly labeled as "in /etc/hosts on your machine".

Do not link to `.planning/` paths — those are workflow artifacts and `docs/setup.md` is a product-facing doc.
