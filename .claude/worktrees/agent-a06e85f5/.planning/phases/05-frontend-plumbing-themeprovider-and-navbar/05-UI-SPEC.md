# Phase 5 тАФ UI Design Contract

> Visual and interaction contract for Phase 5 (Frontend Plumbing тАФ ThemeProvider and NavBar). Generated by gsd-ui-researcher from CONTEXT.md (D-01..D-19), REQUIREMENTS.md (BRAND-03, BRAND-06), and existing frontend inspection. Verified by gsd-ui-checker.

**Phase scope reminder:** This phase delivers plumbing only тАФ a `<ThemeProvider>` gate, a `useSettings()` hook, the NavBar brand slot swap, a Settings gear entry point, and a `/settings` route stub. No Settings page UI, no color pickers, no live-preview mutation path (those are Phase 6). The design contract below therefore focuses narrowly on the three user-visible surfaces this phase actually touches: the neutral loading skeleton, the NavBar, and the `/settings` stub page.

| Token | Value | Usage in Phase 5 |

| lg | 24px (`gap-6`, `px-6`) | NavBar horizontal inner padding (`max-w-7xl mx-auto px-6`); inter-link gap on the left cluster тАФ **preserve existing** |

- **Settings gear icon:** 20├Ч20 icon (`h-5 w-5`) inside a square ghost Button (default shadcn icon-button size, effective 36├Ч36 touch target тАФ acceptable for desktop internal tool). Grouped with FreshnessIndicator and LanguageToggle on the right (`ml-auto flex items-center gap-4`).

## Typography

Phase 5 touches three strings of user-visible text: the NavBar brand (logo fallback), the existing nav links, and the `/settings` stub page placeholder. No new type roles are introduced. Token values are those already shipped in v1.0 via Tailwind's default scale + Geist Variable.

| Role | Size | Weight | Line Height | Phase 5 Usage |
|------|------|--------|-------------|----------------|
| Nav brand | 14px (`text-sm`) | 600 (`font-semibold`) | 1.25 (`leading-tight`, Tailwind default for `text-sm`) | `<span>{settings.app_name}</span>` when no logo is set (BRAND-06). Replaces the existing `{t('nav.brand')}` span; same class string, different source. |
| Nav link | 14px (`text-sm`) | 400 regular (inactive) / 600 semibold (active) | 1.25 | Existing Dashboard / Upload links; unchanged. |
| Body | 16px (`text-base`) | 400 | 1.5 (`leading-normal`) | `/settings` stub page placeholder body text. |
| Heading | 24px (`text-2xl`) | 600 (`font-semibold`) | 1.2 (`leading-tight`) | `/settings` stub page `<h1>` placeholder (e.g. "Settings"). |

**Rules:**
- Exactly **2 weights** project-wide: 400 regular, 600 semibold. No 500/700 introduced in this phase.
- The NavBar brand slot is **never localized** (D-06): read `settings.app_name` directly, not via `t(...)`. `t('nav.brand')` is deleted from `NavBar.tsx`; the locale key itself stays in `de.json` / `en.json` until Phase 7.
- `document.title` is set to `settings.app_name` unchanged тАФ no suffix, no page-specific title (D-09).

## Color

Phase 5 is a **theming plumbing** phase, not a color-design phase. The 6 semantic tokens already exist as shadcn neutral defaults and are **runtime-overridable** by the user via Phase 6's Settings page. Phase 5's job is to make that override path work end-to-end. The fixed color contract below covers only the surfaces Phase 5 itself renders: the loading skeleton (which must stay neutral and unbranded) and the fallback/default palette.

### 60 / 30 / 10 split (Phase 5 rendered surfaces)

| Role | Token | Default (from `index.css:58тАУ91`) | Phase 5 Usage |
|------|-------|-----------------------------------|----------------|
| Dominant (60%) | `--background` | `oklch(1 0 0)` (pure white) | Page background behind the skeleton; main app area behind NavBar |
| Secondary (30%) | `--card` (follows `--background` at default) + `--muted` | `oklch(1 0 0)` / `oklch(0.97 0 0)` | NavBar background (`bg-card`) and skeleton shimmer bars (`bg-muted`) |
| Accent (10%) | `--primary` | `oklch(0.205 0 0)` (near-black in neutral preset) | Active nav-link underline + active link text (`text-primary`, `border-primary`) тАФ **already in v1.0**; no new accent usage added in this phase |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` (red) | Sonner error toast on settings-fetch failure тАФ reuses existing toast styling |

**Accent reserved for (explicit list, no expansion):**
1. Active NavBar link text and bottom border (existing v1.0 behavior, preserved).
2. **Nothing else in Phase 5.** The Settings gear icon is rendered in the default foreground color (`text-foreground`), not `text-primary`. Hover state is `hover:text-primary`, matching the existing nav-link hover pattern. No filled/solid accent button is introduced.

### Loading skeleton color rules (D-01, D-02)

The neutral loading skeleton is the highest-stakes surface in this phase because it must **never** leak default brand values before stored settings apply. Strict rules:

| Surface | Allowed | Forbidden |
|---------|---------|-----------|
| Full-screen background | `oklch(0.97 0 0)` (the `--muted` default) | Any hardcoded non-neutral color; any use of `--primary`, `--accent`, `--destructive` |
| Skeleton bars / spinner | `bg-muted`, `text-muted-foreground`, shadcn/tw-animate-css `animate-pulse` | Any branded fill, any logo, any app-name text |
| Text content | **Zero text.** No "LoadingтАж", no "KPI Light", no translated strings | Any translated string via `t(...)` (would show `nav.brand` default) |
| Animation | Optional subtle `animate-pulse` on 1тАУ3 centered skeleton bars, OR a lucide `<Loader2 className="animate-spin" />` in `text-muted-foreground` | Branded spinner variants |

Rationale: Success Criterion #1 demands "no default-brand flash during settings fetch." The skeleton is the only thing users see during the fetch, so it must be provably brand-free. Keep it text-free and neutral-only тАФ this is the cheapest way to pass the checker's visual audit.

### Fetch-error fallback (D-03)

On `GET /api/settings` failure, ThemeProvider applies the frontend-side default palette from `frontend/src/lib/defaults.ts` (mirrors `backend/app/defaults.py`), renders the app normally, and fires a single sonner toast. The fallback palette **is** the v1.0 shipped defaults тАФ users never see the skeleton persist; they see a functional app plus an error toast explaining the degraded state. No destructive/red surfaces beyond the toast itself.

## Copywriting Contract

Phase 5 introduces very few new strings. Nearly all existing NavBar copy is preserved; the only **removal** is the `t('nav.brand')` call site (its source data moves to `settings.app_name`). New strings needed:

### New locale keys (add to both `de.json` and `en.json`)

| Key | DE | EN | Where used |
|-----|----|----|------------|
| `nav.settings` | `Einstellungen` | `Settings` | `aria-label` + tooltip on the NavBar gear icon button; accessible name for screen readers |
| `theme.error_toast` | `Einstellungen konnten nicht geladen werden тАФ Standardwerte werden angezeigt.` | `Could not load settings тАФ showing defaults.` | Sonner error toast on settings-fetch failure (D-03) |
| `settings.page_title_stub` | `Einstellungen` | `Settings` | `<h1>` on the `/settings` stub page (Phase 6 may reuse or replace) |
| `settings.stub_body` | `Diese Seite wird in Phase 6 verf├╝gbar.` | `This page will be available in Phase 6.` | Placeholder body copy on the stub page so it is not visually empty during Phase 5's human verification |

### Primary CTA (Phase 5 has none)

| Element | Copy |
|---------|------|
| Primary CTA | **None.** Phase 5 is infrastructure тАФ there is no user action to take on any Phase 5 surface. The Settings gear is **navigation**, not a CTA, and is labeled by its aria-label (`Einstellungen` / `Settings`), not a visible button label. |
| Empty state heading | **Not applicable.** No data-driven surfaces. |
| Empty state body | **Not applicable.** |
| Loading state | **No text.** The skeleton is intentionally text-free (see Color тЖТ skeleton rules). |
| Error state | Sonner toast `theme.error_toast` above. Single toast, single sentence, DE + EN. No inline error panel тАФ the app renders normally with default palette so there is no surface that needs inline error copy. |
| Destructive confirmation | **None.** Phase 5 has zero destructive actions (no delete, no reset, no clear тАФ all deferred to Phase 6). |

### Preserved v1.0 strings (unchanged by this phase)

- `nav.dashboard` (Dashboard / Dashboard)
- `nav.upload` (Upload / Upload)
- `nav.brand` тАФ **call site removed** in `NavBar.tsx`; **JSON key left intact** (D-18). Phase 7 decides whether to delete it.

### Style rules

- **Brand text is not localized.** `settings.app_name` is rendered as a single raw string regardless of current `i18n.language`. This is intentional: a corporate app name is a proper noun, not a translation target.
- **No "KPI Light" literal** anywhere in Phase 5 source code (except the `nav.brand` JSON key values, which remain as a Phase 7 cleanup item). The default app name lives in `frontend/src/lib/defaults.ts` (D-16) and is the single source for the error-fallback path.
- **Settings gear labeling:** The icon button MUST have an `aria-label={t('nav.settings')}` тАФ a lucide `<Settings />` icon alone is not accessible without a programmatic label. A tooltip is optional (Claude's discretion per CONTEXT.md) but the aria-label is non-optional.

Authoritative list of components this phase adds or edits. Downstream planner uses this to shape tasks. No component outside this list is touched.

| `NavBar` | **edit** | `frontend/src/components/NavBar.tsx` | (a) Replace `{t('nav.brand')}` with logo-or-text slot; (b) add right-side gear icon button wrapped in `<Link href="/settings">`; (c) call `useSettings()` to read `app_name` and `logo_url`. |
| `App.tsx` | **edit** | `frontend/src/App.tsx` | Wrap `<NavBar />` + `<main>` in `<ThemeProvider>`; add `<Route path="/settings" component={SettingsPage} />`. `Toaster` stays **outside** `ThemeProvider` so error toasts can fire even if the provider is in its fallback path. |

| lucide `Loader2` icon | reuse | `lucide-react` package | Optional skeleton spinner per D-02. Color `text-muted-foreground`. |

### Component interaction rules

- **Logo slot is mutually exclusive with the text fallback (D-05).** Never render both. Logic: `settings.logo_url != null ? <img> : <span>`. Treat empty string as text fallback (defensive), but per D-03/specifics the backend only ever returns `null` or a populated URL.
- **Logo alt text** is `settings.app_name` тАФ not a translated string, not "Logo". This gives screen readers a meaningful label when the image loads and when it fails to load.
- **Logo `<img>` uses the backend-provided URL verbatim.** The backend appends `?v=<logo_updated_at>` (BRAND-04 already satisfied); the frontend does not add its own cache-buster.
- **ThemeProvider is the single writer of `:root` inline styles (D-15).** Phase 6's live-preview mutation writes through `queryClient.setQueryData(['settings'], draft)` which invalidates ThemeProvider's `useEffect` dependency тАФ NOT by calling `setProperty` from the Settings page.
- **`document.title` mirrors `settings.app_name`** (D-09). No suffix. ThemeProvider's same `useEffect` handles it so the title flips alongside colors in a single re-render.

## Interaction States

| State | Trigger | Visual | Copy |
|-------|---------|--------|------|
| **Initial load (pending)** | `useSettings().isLoading === true` | Full-screen neutral skeleton on `bg-muted`, optional centered `Loader2` spinner or 2тАУ3 skeleton bars, `animate-pulse` | **No text** (D-02) |
| **Success** | `useSettings().data` available | Full app renders: NavBar with logo-or-name, routed page body, live CSS vars applied | Normal app content; `document.title = settings.app_name` |
| **Error fallback** | `useSettings().error !== null` | App renders with **default** palette from `defaults.ts`, default app name in NavBar, single sonner toast | Toast copy = `theme.error_toast` (1 line, DE + EN) |
| **NavBar gear hover** | Pointer over gear button | Ghost button bg transitions to `bg-accent/10` (shadcn default) | Tooltip optional; `aria-label` always present |
| **NavBar gear active route** | `location === '/settings'` | Gear icon color = `text-primary`; optionally a 2px `border-b-2 border-primary` echoing the existing nav-link active pattern | Same `aria-label` |
| **Logo image load error** | `<img>` `onError` (broken URL, 404) | Fall back to text span rendering `settings.app_name` in the same `text-sm font-semibold` class | No toast тАФ logo URL mismatch is silent; stale browser cache will self-heal on next settings fetch |

**The logo `<img onError>` fallback is a recommended safety net**, not strictly required by BRAND-03 (which only covers the `logo_url === null` case). Downstream planner may decide whether to implement it; the UI contract permits either path.
