  <what-built>
    ThemeProvider is mode-aware with MutationObserver (Plan 01), Recharts charts consume chartDefaults with full token coverage (Plan 02), consumer components use token-based utilities instead of hardcoded Tailwind colors (Plan 03).
  </what-built>
  <how-to-verify>
    1. Start the app:
       - From repo root: `docker compose up -d` (if using Docker) OR `cd frontend && npm run dev`
       - Open http://localhost:5173 (Vite default) or the configured URL.

    2. Open browser devtools Console. Keep it visible.

    3. **Light mode baseline:**
       - Visit Sales dashboard, HR dashboard, Upload page, Settings page.
       - Note the current brand primary color (e.g., the default blue on primary buttons / current-period chart line).

    4. **Toggle dark mode:**
       - In Console: `document.documentElement.classList.add('dark')`.
       - Expected: every page surface inverts — dark backgrounds (`oklch(0.145 0 0)` body, `oklch(0.205 0 0)` cards), light text.

    5. **Per-page checks in dark mode:**

       **Sales dashboard:**
       - [ ] KPI cards: dark card background, light foreground numbers, borders visible but subtle
       - [ ] Revenue chart: dark bg, light axis tick labels, readable tooltip on hover (dark popover + light text), current-period line/bar uses brand primary, prior-period overlay is amber
       - [ ] Legend text readable
       - [ ] Date-range filter, segmented controls: readable, brand-primary active state

       **HR dashboard:**
       - [ ] HR KPI cards, HR charts: same checks as Sales
       - [ ] EmployeeTable: active-status badge uses success-green tint with readable foreground; other rows use muted bg/text

       **Upload page:**
       - [ ] DropZone idle: muted bg, subtle border
       - [ ] DropZone drag-active (drag a file over): primary border, primary/5 bg, primary text
       - [ ] DropZone invalid file: destructive red text
       - [ ] Browse button: brand-primary background (default shadcn variant)
       - [ ] UploadHistory: status badges readable — success green, warning amber, failed destructive
       - [ ] Delete-row hover: destructive red text on destructive/10 bg

       **Settings page:**
       - [ ] Form inputs, labels, cards all readable on dark bg
       - [ ] Personio card: success indicator is success-green token
       - [ ] Color-picker preview swatches: still show literal brand colors (NOT themed — this is the intentional exception per UI-SPEC)

       **Dialogs / Toasts / Popovers:**
       - [ ] Delete-confirm dialog: dark overlay (bg-black/10 from shadcn is acceptable), dialog itself is dark card with readable text
       - [ ] Any open popover/tooltip: dark popover bg, light popover-foreground text

    6. **Brand accent invariance check (DM-04):**
       - In Console, with `.dark` active:
         `getComputedStyle(document.documentElement).getPropertyValue('--primary')`
       - Remove `.dark`: `document.documentElement.classList.remove('dark')`
       - Run same getComputedStyle again.
       - EXPECT: both values identical strings. If they differ, ThemeProvider is leaking.

    7. **Amber invariance check (D-09):**
       - With `.dark` active:
         `getComputedStyle(document.documentElement).getPropertyValue('--color-warning')`
       - Remove `.dark`, run again.
       - EXPECT: `#facc15` in both cases.

    8. **Recharts inline-hex audit:**
       - Run: `grep -rnE "(fill|stroke|color|background)=['\"]#" frontend/src/components/`
       - EXPECT: empty output.

    9. **Full-frontend hardcoded-utility audit:**
       - Run: `grep -rnE "\\b(bg|text|border)-(gray|slate|neutral|zinc|red|green|blue|yellow|amber)-[0-9]+\\b" frontend/src --include='*.tsx' --include='*.ts' | grep -v "index.css"`
       - EXPECT: only `dialog.tsx` `bg-black/10` match (documented exception) — or empty.

    10. Toggle `.dark` off, confirm the app returns to the brand light palette with no residual dark styling.
