# Phase 30.1 Context — Rebrand KPI Light → KPI Dashboard

## Phase Goal

Change the human-facing product name from "KPI Light" to "KPI Dashboard" across UI, documentation, and defaults — without touching any internal identifier (OIDC client IDs, compose service names, docker image names, repo directory, Dex config, redirect URIs).

## Domain Boundary

**In scope (display name only):**
- Frontend i18n values in `en.json` / `de.json` (`common.appName`, any other string literal with "KPI Light")
- Frontend default settings (`frontend/src/lib/defaults.ts` + `frontend/src/bootstrap.ts`)
- Backend `DEFAULT_SETTINGS.app_name` in `backend/app/defaults.py`
- Alembic migration updating existing `app_settings` row where `app_name = 'KPI Light'`
- `README.md` product description + title
- `docs/setup.md` narrative references (NOT hostnames or compose service names)
- `CLAUDE.md` project name line
- `.planning/PROJECT.md` living doc: add "Renamed from KPI Light on 2026-04-15" note + update title/description
- `docker-compose.yml` narrative comments only (do NOT rename services)

**Out of scope (internal identifiers — DO NOT TOUCH):**
- Repo directory `acm-kpi-light` (would break `docker compose` project name)
- Docker compose service names (`api`, `frontend`, `db`, etc. — already generic)
- Docker image names (`acm-kpi-light-api`, `acm-kpi-light-frontend`, `acm-kpi-light-migrate` — compose-derived from directory)
- Dex OIDC client ID `kpi-light` in `dex/config.yaml` (changing breaks live sessions + requires .env re-keying + Outline cares too)
- `DEX_KPI_SECRET` env var name
- Redirect URI hostname `kpi.internal` (operator runs DNS)
- The 95 historical `.planning/` artifacts from prior phases — leave as-is (records of decisions made under the old name)
- OIDC scope `kpi-light` in source code identifiers
- Database names, table names, DB container name `db`

## Carried-Forward Decisions

- **PROJECT.md is the living doc for project identity** (Phase 26-29 precedent) — treat it as the authoritative name source and update it; historical prose in earlier sections may keep "KPI Light" since those entries describe events that happened under that name.
- **Dual flat+nested i18n keys** (Phase 28 D) — for any new/moved keys; this phase changes **values**, not keys, so the pattern is preserved automatically.
- **Alembic for DB schema + data changes** — required by CLAUDE.md; migration file follows the `v1_NN_description.py` pattern, down_revision chained from current head.
- **Image pins + healthcheck-gated startup** — no compose service changes; existing patterns intact.

## Decisions (Phase 30.1)

### D-01: Display name only — internal identifiers untouched
New display name: **`KPI Dashboard`** (two words, title case).
Internal identifiers stay: `kpi-light` (Dex client), `acm-kpi-light` (repo/compose project), `kpi.internal` (hostname), `DEX_KPI_SECRET` (env var).

**Rationale:** Minimizes blast radius. Rename is 30-min cosmetic work vs. a half-day migration if internal IDs move. The inconsistency between display name and identifiers is acceptable for an internal tool; a future deep-rename phase can tackle identifiers if the operator ever cares.

### D-02: Historical `.planning/` artifacts stay untouched
No find-replace across `.planning/phases/*` / `REQUIREMENTS.md` / `ROADMAP.md` entries written under the old name. The historical record reflects the state at the time each artifact was authored.

**Exception:** `.planning/PROJECT.md` IS updated — it's a living document. Add a dated rename line to its `## Current State` / shipped-versions section.

**Rationale:** Preserves traceability + git blame. Downstream agents reading past CONTEXT/SUMMARY files are explicitly asked below to interpret "KPI Light" in those files as referring to the same product.

### D-03: i18n — change values, keep keys
- `en.json`: `common.appName: "KPI Light"` → `common.appName: "KPI Dashboard"`
- `de.json`: same change (German UI uses the same proper noun by convention)
- All other literal `"KPI Light"` strings in locale files switch to `"KPI Dashboard"` (grep-audit during execution)
- Keys remain unchanged → zero downstream `t("common.appName")` callsite churn

**Rationale:** Decouples wording change from code churn. Safer refactor.

### D-04: DB migration for existing `app_settings.app_name`
Alembic migration `v1_11_rename_app_name.py` (chained after the current head):

```python
def upgrade():
    op.execute("""
        UPDATE app_settings
        SET app_name = 'KPI Dashboard'
        WHERE app_name = 'KPI Light'
    """)

def downgrade():
    op.execute("""
        UPDATE app_settings
        SET app_name = 'KPI Light'
        WHERE app_name = 'KPI Dashboard'
    """)
```

Also update `DEFAULT_SETTINGS.app_name` in `backend/app/defaults.py` to `"KPI Dashboard"` so fresh installs and `/settings/reset-defaults` land on the new name.

**Rationale:** Operators who haven't customized `app_name` see the rename automatically. Operators who did (e.g. branded the settings page) are unaffected because the WHERE clause only touches the default value.

### D-05: Commit discipline + PROJECT.md history note
- One commit per logical file change (compose narrative, backend defaults, migration, frontend defaults + i18n, README, docs/setup.md, CLAUDE.md, PROJECT.md)
- Commit messages: `docs(30.1): rebrand {component} KPI Light → KPI Dashboard` / `feat(30.1): migrate app_settings.app_name default`
- Final commit updates PROJECT.md:
  - Title: "KPI Dashboard"
  - Add sentence to Core Value / description: "Formerly called KPI Light (renamed 2026-04-15, Phase 30.1)."

**Rationale:** Reversible + reviewable; if a specific rename commit turns out to be wrong it's isolable.

### D-06: Phase 31 CONTEXT/plans get a follow-up edit (NOT in this phase)
At the end of Phase 30.1 execution, a **thin fix-up task** inside 30.1 will update the three Phase 31 references ("KPI Light collection" → "KPI Dashboard collection") in:
- `.planning/phases/31-seed-outline-docs/31-CONTEXT.md` (decisions, canonical refs)
- `.planning/phases/31-seed-outline-docs/31-01-PLAN.md` through `31-06-PLAN.md` (collection name in tasks)
- `.planning/ROADMAP.md` Phase 31 entry (Goal + Success Criteria)

This is an exception to D-02 because Phase 31 has NOT executed yet — the collection doesn't exist; there's no history to preserve.

**Rationale:** Avoid authoring 8 wiki pages and then renaming the collection mid-seed.
