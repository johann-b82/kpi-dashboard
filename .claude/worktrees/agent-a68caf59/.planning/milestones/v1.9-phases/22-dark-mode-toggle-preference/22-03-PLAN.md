  <how-to-verify>
    Open the dev server URL. For each scenario, open browser DevTools → Application → Local Storage and clear the `theme` key BEFORE starting the scenario (unless the scenario specifies otherwise). Test on whichever OS the user develops on (macOS: System Settings → Appearance; Windows: Settings → Personalization → Colors).

    **Scenario A — DM-05 (toggle visible & functional):**
    - [ ] Toggle appears in navbar to the LEFT of the DE/EN toggle
    - [ ] Visual size, padding, border-radius match the DE/EN toggle (sibling appearance)
    - [ ] Active segment is highlighted with `bg-primary` (the brand accent), inactive segment is muted
    - [ ] Clicking "Dark" immediately darkens entire app (cards, charts, tables, forms)
    - [ ] Clicking "Light" immediately lightens entire app
    - [ ] Brand accent color in highlighted segment is identical in both modes (DM-04 invariance preserved)

    **Scenario B — DM-06 (OS preference on first visit):**
    - [ ] Set OS to LIGHT mode. Clear localStorage. Hard reload (Cmd/Ctrl+Shift+R). App is in light mode and "Light" segment is selected.
    - [ ] Set OS to DARK mode. Clear localStorage. Hard reload. App is in dark mode and "Dark" segment is selected.
    - [ ] **Live OS tracking (no stored value):** Clear localStorage. Reload. With the page open, change OS theme in System Settings. App switches mode within ~1 second AND the active segment updates to match.

    **Scenario C — DM-07 (localStorage persistence overrides OS):**
    - [ ] Set OS to LIGHT mode. Click "Dark" in toggle. Hard reload. App is in dark mode (localStorage wins, NOT OS).
    - [ ] Set OS to DARK mode. Click "Light" in toggle. Hard reload. App is in light mode (localStorage wins, NOT OS).
    - [ ] DevTools → Application → Local Storage → `theme` key value matches the clicked mode (literally `dark` or `light`, no JSON quotes).
    - [ ] **Locked after first click:** With localStorage set to "dark" and app in dark mode, change OS to light. App stays dark (matchMedia listener no-ops because stored value exists).
    - [ ] **Manual tampering fallback (D-11):** In DevTools, set `localStorage.theme = "banana"`. Hard reload. App falls back to OS preference; no console error thrown.

    **Scenario D — DM-08 (i18n parity):**
    - [ ] EN: segments show "Light" and "Dark"
    - [ ] DE: segments show "Hell" and "Dunkel"
    - [ ] DevTools → inspect the `<div role="radiogroup">` — `aria-label="Theme"` in EN, `aria-label="Farbschema"` in DE
    - [ ] Switching language via the DE/EN toggle does NOT change the active theme
    - [ ] Switching theme does NOT change the active language

    **Scenario E — FOUT prevention (success criterion 4 + D-04/D-05):**
    - [ ] Set OS to LIGHT, click "Dark" in app, hard reload (Cmd/Ctrl+Shift+R). At first paint, app appears DARK — no white flash before darkening. (Throttle Network to "Slow 3G" in DevTools to make any flash visible if needed.)
    - [ ] Set OS to DARK, click "Light" in app, hard reload. At first paint, app appears LIGHT — no dark flash.
    - [ ] Acceptable known limitation: the bootstrap-splash element (`#root > .bootstrap-splash`) uses hardcoded `#ffffff` background — pre-existing from before Phase 22, OUT OF SCOPE here, belongs to Phase 23 contrast audit. Do NOT fail the gate on it.

    **Scenario F — Cross-page consistency:**
    - [ ] Navigate to /, /hr, /upload, /settings — current theme persists across all routes
    - [ ] Theme toggle remains visible and functional on every route except where the back-arrow replaces nav (settings/upload pages — verify `<ThemeToggle />` is still in the right-side action group)

    **Report:**
    - If ALL boxes ticked → respond `approved`
    - If ANY scenario fails → respond `Scenario X: {one-line description}` per failure
