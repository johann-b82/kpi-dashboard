Extend ThemeProvider to be mode-aware so brand surface colors stop stomping the shadcn `.dark` CSS block in dark mode, while brand accent/primary remain applied in both modes (DM-04).

Purpose: Without this, toggling `.dark` has no visible effect because inline `style.setProperty` calls beat class-rule specificity. This is the load-bearing fix for DM-01 and DM-04.
Output: Mode-aware ThemeProvider that reacts to `.dark` class changes on `<html>` via MutationObserver.
</objective>

  <behavior>
    - In light mode (no `.dark` class on html): applyTheme sets all 6 tokens via root.style.setProperty
    - In dark mode (`.dark` class present on html): applyTheme sets only --primary and --accent inline; removes inline --background, --foreground, --muted, --destructive
    - When `.dark` is added to html externally: observer fires, applyTheme re-runs, surface inline styles are removed
    - When `.dark` is removed from html externally: observer fires, applyTheme re-runs, surface inline styles are re-applied from Settings
    - Cleanup on unmount: observer.disconnect() called
  </behavior>
  <action>
    Modify `frontend/src/components/ThemeProvider.tsx`:

    1. Define two const arrays at module scope (above the component), typed to keys of THEME_TOKEN_MAP:
    ```typescript
    const SURFACE_TOKEN_KEYS = [
      'color_background',
      'color_foreground',
      'color_muted',
      'color_destructive',
    ] as const satisfies ReadonlyArray<keyof typeof THEME_TOKEN_MAP>;

    const ACCENT_TOKEN_KEYS = [
      'color_primary',
      'color_accent',
    ] as const satisfies ReadonlyArray<keyof typeof THEME_TOKEN_MAP>;
    ```

    2. Replace the existing `applyTheme` function body so it:
       - Reads `const isDark = root.classList.contains('dark');` (per D-03)
       - Always applies ACCENT_TOKEN_KEYS via `root.style.setProperty(THEME_TOKEN_MAP[key], settings[key])` (per D-02, DM-04)
       - If `isDark`: iterate SURFACE_TOKEN_KEYS and call `root.style.removeProperty(THEME_TOKEN_MAP[key])` (per D-01, D-03)
       - Else: iterate SURFACE_TOKEN_KEYS and call `root.style.setProperty(THEME_TOKEN_MAP[key], settings[key])`
       - Keep the existing `document.title = settings.app_name;` line (or equivalent) if it already lives in applyTheme

    3. Inside the existing `useEffect` that calls `applyTheme(effective)`, after the initial call, set up a MutationObserver (per D-14):
    ```typescript
    const observer = new MutationObserver(() => {
      applyTheme(effective);
    });
    observer.observe(document.documentElement, {
      attributes: true,
      attributeFilter: ['class'],
    });
    return () => observer.disconnect();
    ```
    The `effective` dependency in the useEffect dependency array ensures the observer's closure stays fresh.

    4. Do NOT remove or change any other ThemeProvider behavior (settings fetching, context value, provider JSX). Implements D-01, D-02, D-03, D-04 and contributes to DM-01, DM-04.
  </action>
