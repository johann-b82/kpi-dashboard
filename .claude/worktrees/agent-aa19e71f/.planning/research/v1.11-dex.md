# Dex OIDC Research — KPI Light v1.11

**Researched:** 2026-04-14
**Confidence:** MEDIUM — version pinning HIGH, config patterns MEDIUM (docs + community sources), FastAPI integration MEDIUM

---

## Summary Verdict

Dex is the right choice for this use case. It is a single static Go binary, ships as a tiny Docker image (~50 MB), and is purpose-built for exactly this scenario: a small cluster of apps sharing one OIDC issuer, with users defined in config. The main operational trade-off is that user management requires editing `config.yaml` and restarting the container — there is no admin UI. For 5–10 internal users who change infrequently, that is acceptable.

The critical integration gotcha is the **issuer URL**: it must be reachable by the user's **browser** (for the OAuth2 redirect loop), and that same URL is embedded in every ID token as the `iss` claim. Backend containers (FastAPI) must also be able to reach it to fetch JWKS/discovery. On a single server behind a reverse proxy, the cleanest pattern is: set `issuer` to the public HTTPS URL (e.g. `https://auth.example.com`), and let the reverse proxy handle TLS termination. Do not use an internal Docker network hostname as the issuer.

## 1. Current Version and Docker Image

**Latest stable release:** `v2.43.0` (released 2025-05-19)
**Previous stable:** `v2.42.1` (released 2025-04-14), `v2.42.0` (released 2025-02-19)

**Recommended image pin:**
```
ghcr.io/dexidp/dex:v2.43.0
```
The image is published on both Docker Hub (`dexidp/dex`) and GitHub Container Registry (`ghcr.io/dexidp/dex`). Prefer GHCR as Docker Hub rate-limits unauthenticated pulls. The `-alpine` variant exists but the default image is already Alpine-based and minimal.

**Known CVE note (LOW confidence — unverified from official advisory):**
Two CVEs (CVE-2025-68121, CVE-2026-25934) were found in the `gomplate` binary bundled in the Dex image. Gomplate is used only for config pre-processing and is optional — if you use a plain `config.yaml` without gomplate templating, the binary is never invoked. Fixes are pending upstream gomplate. Monitor the Dex releases page for a patched image.

**Compatibility:**
- Dex v2.x is the current major line — v3 does not exist as of this research.
- The config format has been stable since v2.28; no breaking changes expected between v2.40 and v2.43.
- SQLite3 storage is fully supported in v2.43 and has been production-usable since at least v2.35.

## 2. Minimum Viable Config

### docker-compose service snippet

```yaml
services:
  dex:
    image: ghcr.io/dexidp/dex:v2.43.0
    container_name: dex
    restart: unless-stopped
    command: dex serve /etc/dex/config.yaml
    ports:
      # Expose on host so the reverse proxy can reach it.
      # Browser redirects go through the reverse proxy, not directly.
      - "5556:5556"
    volumes:
      - ./dex/config.yaml:/etc/dex/config.yaml:ro
      - dex_data:/data
    environment:
      # Pass secrets via env vars; reference them in config.yaml with $VAR syntax
      # (gomplate not required — dex natively supports env var substitution in config)
      DEX_KPI_SECRET: "${DEX_KPI_SECRET}"
      DEX_OUTLINE_SECRET: "${DEX_OUTLINE_SECRET}"
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:5556/dex/healthz"]
      interval: 15s
      timeout: 5s
      retries: 3
    depends_on: []   # Dex has no DB dependency when using SQLite

volumes:
  dex_data:
```

Add to `.env`:
```
DEX_KPI_SECRET=change-me-kpi-light-client-secret
DEX_OUTLINE_SECRET=change-me-outline-client-secret
```

### Example config.yaml

```yaml
# /dex/config.yaml
# Dex v2.43.x — static-password connector, SQLite storage, two OIDC clients

issuer: https://auth.example.com  # MUST be the public-facing URL. See Section 5.

storage:
  type: sqlite3
  config:
    file: /data/dex.db            # Persisted via the dex_data volume mount.

web:
  http: 0.0.0.0:5556              # Dex listens on plain HTTP; TLS is terminated at reverse proxy.

telemetry:
  http: 0.0.0.0:5558              # Optional metrics endpoint; can be omitted.

grpc:                             # Optional: enables the gRPC API for programmatic user ops.
  addr: 127.0.0.1:5557
  reflection: true

logger:
  level: "info"
  format: "json"

# ---- OAuth2 behavior ----
oauth2:
  skipApprovalScreen: true        # Internal tool — no need for the "Grant access?" consent page.
  responseTypes:
    - code                        # Authorization Code flow only. Do NOT add "token" — that's implicit flow.
  grantTypes:
    - authorization_code
    - refresh_token

# ---- Expiry / session config ----
expiry:
  deviceRequests: "5m"
  signingKeys: "6h"
  idTokens: "24h"                 # How long the ID token JWT is valid after issuance.
                                  # 24h is reasonable for an internal tool; lower (1h) if more security needed.
  refreshTokens:
    validIfNotUsedFor: "720h"     # 30 days — refresh token expires if not used for 30 days.
    absoluteLifetime: "2160h"     # 90 days — hard cap regardless of use.
    reuseInterval: "3s"           # Grace window for concurrent requests using same refresh token.
    disableRotation: false        # Keep rotation on; safer default.

# ---- Static password connector (replaces upstream LDAP/OAuth) ----
enablePasswordDB: true

staticPasswords:
  - email: "alice@example.com"
    # Generate: htpasswd -bnBC 10 "" 'YourPassword' | tr -d ':\n'
    # Or:       python3 -c "import bcrypt; print(bcrypt.hashpw(b'pass', bcrypt.gensalt(10)).decode())"
    hash: "$2a$10$REPLACE_WITH_REAL_BCRYPT_HASH_ALICE"
    username: "alice"
    userID: "c89b3d7e-0001-0000-0000-000000000001"   # Stable UUID; generate once, never change.

  - email: "bob@example.com"
    hash: "$2a$10$REPLACE_WITH_REAL_BCRYPT_HASH_BOB"
    username: "bob"
    userID: "c89b3d7e-0002-0000-0000-000000000002"

# ---- OIDC clients ----
staticClients:
  - id: kpi-light
    secret: $DEX_KPI_SECRET       # Env var substitution — no quotes around $VAR.
    name: "KPI Light"
    redirectURIs:
      - https://kpi.example.com/auth/callback   # FastAPI OIDC callback endpoint.
      - http://localhost:8000/auth/callback      # Dev only — remove in production.

  - id: outline
    secret: $DEX_OUTLINE_SECRET
    name: "Outline Wiki"
    redirectURIs:
      - https://wiki.example.com/auth/oidc.callback   # Outline's expected path.
      - http://localhost:3000/auth/oidc.callback       # Dev only — remove in production.
```

**Notes on this config:**
- The `$VAR` substitution in `config.yaml` values is supported natively by Dex (no gomplate needed).
- `userID` must be a stable, unique identifier. Generate a UUID once and hardcode it — changing it breaks existing sessions/tokens for that user.
- Bcrypt cost 10 is fine; cost 12 is slightly more secure but adds ~300ms to each login.

## 3. User Management Realities

### How it works

Dex's static password connector stores users in `config.yaml`. There is **no admin UI** and **no REST API for user management** (the gRPC API can manage users when the local connector is active, but requires building a gRPC client from the proto definitions — not a point-and-click experience).

**Adding a user:**
1. Generate a bcrypt hash: `python3 -c "import bcrypt; print(bcrypt.hashpw(b'newpassword', bcrypt.gensalt(10)).decode())"`
2. Append to `staticPasswords` in `config.yaml` with a new UUID.
3. `docker compose restart dex` (< 2 seconds — Dex starts instantly).
4. Optionally: `docker compose exec dex dex version` to verify it came back cleanly.

**Removing a user:**
1. Delete their entry from `staticPasswords`.
2. Restart Dex.
3. **Caveat:** Existing refresh tokens for that user in SQLite are not automatically invalidated on restart. They will fail on next use when Dex can't find the user, effectively logging them out within the `validIfNotUsedFor` window. For immediate forced logout, clear `dex_data` volume (nuclear) or use the gRPC API to delete the offline session record.

**Verdict for 5–10 users:** Acceptable. The restart is fast and the config is readable. Document the bcrypt command in the team runbook. If user churn becomes frequent (> a few per month), revisit — the gRPC API path is the escape hatch before switching storage backends.

### gRPC API (optional, advanced)

Enable by adding the `grpc` block to config (shown above). The API supports creating/updating/deleting passwords and offline sessions. Protobuf schema is at `api/v2/api.proto` in the Dex source. No off-the-shelf web UI wraps it — you would need to write a small CLI or use `grpcurl`.

## 4. Session and Token Lifetime

### Recommended defaults for internal tools

| Setting | Recommended | Rationale |
|---------|-------------|-----------|
| `expiry.idTokens` | `24h` | Long enough to not annoy users; short enough to pick up user deletions within a day. |
| `expiry.refreshTokens.validIfNotUsedFor` | `720h` (30 days) | User won't be logged out if they use the tool regularly. |
| `expiry.refreshTokens.absoluteLifetime` | `2160h` (90 days) | Hard cap — forces re-login quarterly. |
| `expiry.signingKeys` | `6h` (default) | Key rotation interval; transparent to users. |
| `oauth2.skipApprovalScreen` | `true` | Eliminates the consent step every internal user finds confusing. |

### How refresh tokens work in Dex

Dex issues a refresh token alongside the ID token when the `offline_access` scope is requested. The refresh token is stored in SQLite (encrypted). When the ID token expires, the OIDC client presents the refresh token to Dex's `/token` endpoint to get a new ID token without user interaction. Dex rotates the refresh token on each use by default (`disableRotation: false`), invalidating the previous one.

**Implication for FastAPI:** Your OIDC client library (Authlib recommended — see Section 6) must store and manage refresh tokens if you want seamless background re-authentication. For a simple "login, get session cookie, stay logged in for N hours" pattern, you can ignore refresh tokens at the application level and just use session expiry matching the `idTokens` lifetime.

## 5. Networking — Issuer URL Gotcha

This is the most common misconfiguration source.

### The rule

The `issuer` value in `config.yaml` is baked into every ID token as the `iss` claim. OIDC clients (FastAPI, Outline) validate that the `iss` claim matches the issuer URL they were configured with. Browsers **must** be able to reach the issuer URL to complete the authorization code redirect flow.

**Do NOT use:**
- `http://dex:5556` — Docker internal hostname. Browser cannot resolve it.
- `http://localhost:5556` — Works in dev, but breaks when apps run on a remote server.

**Use:**
- `https://auth.example.com` — The public-facing URL, regardless of whether Dex listens on HTTP internally.

### Single-server Docker Compose pattern

```
                  Browser
                     |
                     | HTTPS (443)
                     v
              [ Nginx / Traefik ]
                /              \
        /dex/* path          /kpi/* path
               |                    |
          [ dex:5556 ]       [ fastapi:8000 ]
          (HTTP, internal)   (HTTP, internal)
```

- Nginx terminates TLS, proxies `/dex/` to Dex container on port 5556.
- Dex `issuer` is `https://auth.example.com` (or `https://example.com/dex` if using a path).
- FastAPI and Outline reference the same `https://auth.example.com` issuer for JWKS discovery.
- FastAPI container makes token introspection calls to `http://dex:5556` (internal Docker network), but validates `iss == https://auth.example.com`. This works — the introspection network path and the `iss` claim are independent.

### Path-based issuer (optional)

If you prefer a single hostname with path prefixes (e.g. `https://example.com/dex`), set `issuer: https://example.com/dex` and configure Nginx to proxy `location /dex/` to the Dex container. This is cleaner for single-hostname setups.

### Dev / localhost pattern

For local development without TLS:
```yaml
issuer: http://localhost:5556/dex
```
Set `OIDC_ISSUER=http://localhost:5556/dex` in your backend `.env`. Use `http://localhost:8000/auth/callback` as FastAPI's redirect URI. Do not use `dex` as the issuer hostname in dev — your browser won't resolve it.

## 6. TLS / HTTPS

### Production

Dex does **not** require you to configure TLS inside its own config — it is designed to run behind a TLS-terminating reverse proxy. Set `web.http: 0.0.0.0:5556` and let Nginx/Traefik handle HTTPS.

Dex does support native TLS (`web.https` + `web.tlsCert` + `web.tlsKey`) but this is unnecessary complexity when you already have a reverse proxy.

**Do not set** `web.allowedOrigins` unless you are dealing with CORS issues from a SPA making direct calls to Dex — for a server-side OIDC flow (Authorization Code), CORS does not apply.

### Dev / local

Plain HTTP works. Set `issuer: http://localhost:5556/dex` and use the `http` listener. No certs needed.

### Self-signed certs (edge case)

If you must run Dex with a self-signed cert (no reverse proxy), your FastAPI and Outline containers will reject TLS verification. Either:
- Mount the CA cert into the app containers and set `SSL_CERT_FILE` env var, or
- Use plain HTTP on an internal network (acceptable if not internet-facing).

## 7. FastAPI OIDC Integration

### Recommended library: Authlib

```
pip install authlib httpx
```

Authlib has a first-class FastAPI OAuth client. It is actively maintained and handles discovery, token exchange, and JWT validation.

**Basic OIDC flow outline for FastAPI:**

```python
from authlib.integrations.starlette_client import OAuth
from starlette.config import Config
from starlette.middleware.sessions import SessionMiddleware
from fastapi import FastAPI, Request
from fastapi.responses import RedirectResponse

config = Config(".env")
app = FastAPI()
app.add_middleware(SessionMiddleware, secret_key=config("SESSION_SECRET"))

oauth = OAuth(config)
oauth.register(
    name="dex",
    server_metadata_url=f"{config('OIDC_ISSUER')}/.well-known/openid-configuration",
    client_id=config("OIDC_CLIENT_ID"),
    client_secret=config("OIDC_CLIENT_SECRET"),
    client_kwargs={"scope": "openid email profile offline_access"},
)

@app.get("/auth/login")
async def login(request: Request):
    redirect_uri = request.url_for("auth_callback")
    return await oauth.dex.authorize_redirect(request, redirect_uri)

@app.get("/auth/callback")
async def auth_callback(request: Request):
    token = await oauth.dex.authorize_access_token(request)
    user = token.get("userinfo")  # Authlib auto-fetches userinfo
    request.session["user"] = dict(user)
    return RedirectResponse(url="/")

@app.get("/auth/logout")
async def logout(request: Request):
    request.session.clear()
    return RedirectResponse(url="/")
```

Required `.env` entries:
```
OIDC_ISSUER=https://auth.example.com
OIDC_CLIENT_ID=kpi-light
OIDC_CLIENT_SECRET=change-me-kpi-light-client-secret
SESSION_SECRET=random-32-char-secret
```

**JWKS validation note:** Authlib's `authorize_access_token` automatically fetches the JWKS from Dex (`{issuer}/keys`) and validates the ID token signature. You do not need to manually call the userinfo endpoint — `token["userinfo"]` is populated by Authlib when `openid` scope is requested.

**Session middleware note:** `SessionMiddleware` requires `starlette`. It is included with FastAPI — no extra install. Use a strong random `SESSION_SECRET`; this key signs the session cookie.

### Alternative: python-jose for pure JWT validation

If you only need to validate bearer tokens in API routes (not full browser redirect flow), `python-jose[cryptography]` is simpler. Fetch JWKS from `{issuer}/keys`, cache it, validate incoming `Authorization: Bearer <token>` headers. This is the pattern for protecting API endpoints after the browser has already authenticated.

## 8. Fallback Path

If Dex becomes a maintenance burden:

**Option A: Outline native auth + separate FastAPI session**
Outline supports email/password login natively (no IdP required). Enable it in Outline config. For KPI Light, implement a simple FastAPI session layer using username/password stored in the PostgreSQL database (hashed with bcrypt). Two separate auth systems, no shared SSO, but zero external IdP dependency. Complexity: low.

**Option B: Replace Dex with Zitadel**
Zitadel has an admin UI, user management via API, and ships as a single binary/Docker container. It is heavier than Dex but lighter than Authentik. Migration path: same OIDC client config (`client_id`, `client_secret`, redirect URIs) — just update `OIDC_ISSUER` in both apps. Complexity: medium.

**Option C: Caddy + forward-auth**
If the goal is just protecting a URL behind a login wall, Caddy's `forward_auth` directive with a tiny auth service (e.g. `thomseddon/traefik-forward-auth` adapted for Caddy) sidesteps OIDC entirely. Complexity: low, but per-app, not shared identity.

## Integration Gotchas Summary

| Gotcha | Description | Fix |
|--------|-------------|-----|
| Issuer URL browser reachability | `issuer` in config must be DNS-resolvable by the browser. Docker service names (`dex`) are not. | Use public hostname (`https://auth.example.com`) as issuer always. |
| Issuer URL mismatch in token | FastAPI and Outline must configure `OIDC_ISSUER` to exactly match the `issuer` in Dex config, including trailing slash consistency. | `https://auth.example.com` — no trailing slash, everywhere. |
| Static client secret exposure | Dex has no PKCE support for confidential clients — the client secret must be kept secret. It goes in `.env`, never in source control. | Use Docker secrets or env vars; rotate if leaked. |
| Refresh token storage | If FastAPI needs seamless token refresh (no re-login on expiry), it must persist the refresh token in the user session or database. | Store in encrypted session cookie or server-side session store. |
| SQLite concurrency | SQLite handles one write at a time. At 5–10 users with infrequent logins, this is a non-issue. Above ~100 concurrent auth ops, migrate to Postgres. | Use the existing KPI Light PostgreSQL instance for Dex storage if scale becomes a concern: change `storage.type: postgres`. |
| User deletion does not invalidate refresh tokens | Deleting a user from `staticPasswords` and restarting Dex does not synchronously invalidate existing SQLite-stored refresh tokens. | Tokens become invalid on next use (Dex can't find user). For immediate revocation: use gRPC API to delete offline sessions, or wipe `dex.db`. |
| `offline_access` scope required for refresh tokens | Dex only issues a refresh token if the client requests the `offline_access` scope. | Add `offline_access` to `client_kwargs.scope` in Authlib config. |
| Outline redirect URI format | Outline expects the callback at `/auth/oidc.callback` (note the dot). Other paths will fail. | Set `redirectURIs: ["https://wiki.example.com/auth/oidc.callback"]` exactly. |
