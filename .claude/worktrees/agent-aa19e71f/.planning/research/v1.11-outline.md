# Research: Outline Self-Hosted Deployment for KPI Light v1.11

**Researched:** 2026-04-14
**Scope:** Outline wiki in Docker Compose stack with Dex OIDC, Postgres, MinIO-compatible S3, small internal team

## 1. Version + License Verdict

### Current Version

The latest stable release as of April 2026 is **v0.86.0**, with **v1.0.0** released shortly after as a milestone release. Docker Hub tag `0.84` is confirmed published; `0.86` and the `latest` tag point to the most current build.

**Pin recommendation:** `outlinewiki/outline:0.86.0`

Do NOT use `latest` in production — it tracks the rolling release and can introduce breaking migrations unannounced. Monthly cadence means an uncontrolled upgrade could run DB migrations automatically on next container restart.

Notable recent changes (0.82 → 0.86 / 1.0):

## 2. Docker Compose Service Snippet

```yaml
# ─── outline-db ────────────────────────────────────────────────────────────────
outline-db:
  image: postgres:17-alpine
  restart: unless-stopped
  environment:
    POSTGRES_DB: outline
    POSTGRES_USER: outline
    POSTGRES_PASSWORD: ${OUTLINE_DB_PASSWORD}
  volumes:
    - outline_db_data:/var/lib/postgresql/data
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U outline -d outline"]
    interval: 10s
    timeout: 5s
    retries: 5

# ─── outline-redis ─────────────────────────────────────────────────────────────
outline-redis:
  image: redis:7-alpine
  restart: unless-stopped
  volumes:
    - outline_redis_data:/data
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
    interval: 10s
    timeout: 5s
    retries: 5

# ─── outline ───────────────────────────────────────────────────────────────────
outline:
  image: outlinewiki/outline:0.86.0
  restart: unless-stopped
  ports:
    - "3000:3000"         # expose only to reverse proxy; do NOT expose directly
  env_file:
    - ./outline.env
  depends_on:
    outline-db:
      condition: service_healthy
    outline-redis:
      condition: service_healthy
  # If using FILE_STORAGE=local, mount a volume:
  # volumes:
  #   - outline_data:/var/lib/outline/data

volumes:
  outline_db_data:
  outline_redis_data:
  # outline_data:        # uncomment if using local file storage
```

**Notes:**
- Outline runs migrations on startup automatically — no separate migration step required, unlike the KPI Light app's Alembic pattern.
- `condition: service_healthy` is mandatory; without it, Outline starts before Postgres accepts connections and crashes on first boot.
- Keep Outline's Postgres container (`outline-db`) separate from the KPI Light `db` service — different schemas, different migration lifecycles.
- Default internal port is 3000; place behind nginx/Traefik reverse proxy, do not expose to host directly.

## 3. Full Environment Variable Matrix

### Required — MUST set, no sane default

| Variable | Example | Notes |
|----------|---------|-------|
| `SECRET_KEY` | 64-char hex string | Generate: `openssl rand -hex 32`. Rotating this invalidates all sessions. |
| `UTILS_SECRET` | 64-char hex string | Separate from SECRET_KEY. Used for utility endpoints. |
| `DATABASE_URL` | `postgres://outline:${PW}@outline-db:5432/outline` | Must use service name, not `localhost`. |
| `REDIS_URL` | `redis://outline-redis:6379` | Must use service name. |
| `URL` | `https://wiki.internal.example.com` | Full public URL including scheme. No trailing slash. Outline uses this for redirect URIs. |
| `OIDC_CLIENT_ID` | `outline` | Must match client registered in Dex. |
| `OIDC_CLIENT_SECRET` | `<secret>` | Must match Dex static client secret. |
| `OIDC_AUTH_URI` | `https://dex.internal/auth` | Dex authorization endpoint. |
| `OIDC_TOKEN_URI` | `https://dex.internal/token` | Dex token endpoint. Note: use internal DNS name if Outline container reaches Dex container-to-container. |
| `OIDC_USERINFO_URI` | `https://dex.internal/userinfo` | Dex userinfo endpoint. |

### Required — MUST set for S3 storage

| Variable | Example | Notes |
|----------|---------|-------|
| `FILE_STORAGE` | `s3` | Set to `local` to skip S3 entirely (see note below). |
| `AWS_ACCESS_KEY_ID` | `outline-access-key` | MinIO/Garage access key. |
| `AWS_SECRET_ACCESS_KEY` | `<minio-secret>` | MinIO/Garage secret key. |
| `AWS_REGION` | `us-east-1` | Arbitrary string for MinIO/Garage; value doesn't matter, must be set. |
| `AWS_S3_UPLOAD_BUCKET_NAME` | `outline` | Bucket must exist before first start. |
| `AWS_S3_UPLOAD_BUCKET_URL` | `http://minio:9000` | Internal URL of MinIO/Garage container. |
| `AWS_S3_FORCE_PATH_STYLE` | `true` | **Required for MinIO and Garage.** Without this, SDK uses virtual-hosted-style addressing which fails on self-hosted S3. |

### Optional — have sane defaults, but review

| Variable | Default | Notes |
|----------|---------|-------|
| `NODE_ENV` | `production` | Do NOT change. |
| `PORT` | `3000` | Change only if you have a port conflict. |
| `WEB_CONCURRENCY` | `1` | Fine for 5–10 users. Increase to 2–4 for heavier load. |
| `OIDC_DISPLAY_NAME` | `OpenID Connect` | Text on login button. Set to `Dex` or your IdP name. |
| `OIDC_USERNAME_CLAIM` | `preferred_username` | JSON path into JWT payload. Dex emits `preferred_username` by default. |
| `OIDC_SCOPES` | `openid offline_access profile email` | Must include `email` — Outline requires email from OIDC. |
| `OIDC_LOGOUT_URI` | (none) | Dex end-session endpoint: `https://dex.internal/device/logout` or Dex's OIDC discovery endpoint provides it. |
| `FORCE_HTTPS` | `true` | Set to `false` if TLS is terminated at the reverse proxy and Outline is on HTTP internally; otherwise redirect loops occur. |
| `PGSSLMODE` | `disable` | Keep `disable` for container-to-container communication. Enable only if Postgres uses TLS. |
| `AWS_S3_UPLOAD_MAX_SIZE` | `26214400` (25MB) | Max attachment upload size in bytes. |
| `MAXIMUM_IMPORT_SIZE` | `5242880` (5MB) | Max document import size in bytes. |
| `ENABLE_UPDATES` | `true` | Outline phones home to check for updates. Set to `false` for air-gapped deployments. |
| `DEFAULT_LANGUAGE` | `en_US` | Default UI language. |
| `RATE_LIMITER_ENABLED` | `true` | Keep enabled; prevents brute-force on login. |
| `COLLABORATION_URL` | (none) | Only needed if running a separate Collaboration server (not required for small teams). |

### Optional — SMTP

| Variable | Notes |
|----------|-------|
| `SMTP_HOST` | Leave unset to disable email entirely. |
| `SMTP_PORT` | Typically 587 (STARTTLS) or 465 (SSL). |
| `SMTP_USERNAME` | |
| `SMTP_PASSWORD` | |
| `SMTP_FROM_EMAIL` | `noreply@example.com` |
| `SMTP_REPLY_EMAIL` | `noreply@example.com` |
| `SMTP_TLS_CIPHERS` | Only needed for custom TLS config. |
| `SMTP_SECURE` | `true` for port 465. |

**SMTP verdict:** Not required for Outline to function when OIDC handles user provisioning. Without SMTP, invitation emails and document-change notifications are silently skipped. Users can still authenticate via OIDC without any email being sent. Set up SMTP if team wants email notifications; leave unset for a minimal first deployment.

**However:** Outline requires the OIDC provider to return an **email address** in the userinfo/JWT. If Dex is federating from an LDAP/AD source that has no email attribute, you must map or synthesize one in Dex's claims configuration. An empty email claim causes authentication to fail.

## 4. OIDC with Dex — Integration Example

### Dex static client (in `dex-config.yaml`)

```yaml
staticClients:
  - id: outline
    secret: your-outline-oidc-secret   # match OIDC_CLIENT_SECRET
    name: Outline Wiki
    redirectURIs:
      - https://wiki.internal.example.com/auth/oidc.callback
```

### Outline env vars (`outline.env`)

```bash
# Auth — Dex OIDC
OIDC_CLIENT_ID=outline
OIDC_CLIENT_SECRET=your-outline-oidc-secret
OIDC_AUTH_URI=https://dex.internal.example.com/auth
OIDC_TOKEN_URI=https://dex.internal.example.com/token
OIDC_USERINFO_URI=https://dex.internal.example.com/userinfo
OIDC_DISPLAY_NAME=Company SSO
OIDC_USERNAME_CLAIM=preferred_username
OIDC_SCOPES=openid offline_access profile email
# Optional: send user back to Dex logout page on sign-out
# OIDC_LOGOUT_URI=https://dex.internal.example.com/device/logout
```

**Container networking note:** The `OIDC_TOKEN_URI` and `OIDC_USERINFO_URI` are resolved by the Outline **server** container (backend token exchange), not the user's browser. Use internal Docker DNS names (e.g., `http://dex:5556`) for these if Dex is in the same Compose stack. Use the public URL for `OIDC_AUTH_URI` since the user's browser must reach that endpoint. If all endpoints are the same host, be aware of potential internal/external URL split.

**Known Dex quirks:**
1. Dex emits `preferred_username` as the username claim; this is Outline's default so no override needed.
2. Dex requires `offline_access` scope to issue refresh tokens; include it in `OIDC_SCOPES` or sessions expire on short-lived access token TTL.
3. Dex does not expose a standard `end_session_endpoint` in its OIDC discovery document in all configurations. Test `OIDC_LOGOUT_URI` manually; if Dex doesn't support it, omit the variable and users are locally logged out from Outline only.
4. The OIDC callback URL registered in Dex must exactly match `${URL}/auth/oidc.callback` — include `https://` and no trailing slash.

## 5. Storage: Postgres, Redis, S3 — Sizing and What Lives Where

### What lives where

| Store | Contents |
|-------|----------|
| **PostgreSQL** | Document text and structure, user accounts, workspace config, revision history, comments, collection metadata, API tokens |
| **S3 / local storage** | Uploaded image attachments, file attachments, user avatars — binary blobs only |
| **Redis** | Session cache, real-time collaboration state, job queue |

Document content is **not** in S3. A Postgres backup alone recovers all document text. S3 holds only binary attachments. This matters for backup strategy: back up Postgres regularly; S3 attachments can be backed up separately on a slower schedule if needed.

### Sizing for 5–10 users

| Service | Minimum | Comfortable |
|---------|---------|-------------|
| Postgres | 1 vCPU, 512MB RAM, 5GB disk | 1 vCPU, 1GB RAM, 20GB disk |
| Redis | shared CPU, 128MB RAM | 256MB RAM |
| Outline app | 1 vCPU, 512MB RAM | 1 vCPU, 1GB RAM |
| S3 bucket | 1GB initial | scale with attachment usage |

`WEB_CONCURRENCY=1` is appropriate for 5–10 concurrent users. Each worker is a Node.js process; increase to 2 only if you observe latency under load.

### MinIO situation — CRITICAL WARNING

**MinIO community edition Docker images were discontinued in October 2025.** The official `minio/minio` image on Docker Hub is no longer maintained, no longer receives security patches, and contains known vulnerabilities. The maintainers are steering users toward a paid AIStor product.

**Do not use `minio/minio` for new deployments.**

#### Recommended S3 replacement: Garage

[Garage](https://garagehq.deuxfleurs.fr/) is an actively maintained, Rust-based, S3-compatible object store designed for self-hosting. It is:
- Maintained by a non-profit association with EU funding through 2025+
- Significantly lighter than MinIO (runs on a Raspberry Pi)
- Fully S3-compatible for the operations Outline uses (PUT, GET, DELETE objects, presigned URLs)
- Available as a maintained Docker image

**Garage docker-compose snippet:**

```yaml
outline-storage:
  image: dxflrs/garage:v1.0.1
  restart: unless-stopped
  ports:
    - "3900:3900"   # S3 API
    - "3901:3901"   # RPC (internal)
  environment:
    GARAGE_RPC_SECRET: ${GARAGE_RPC_SECRET}
  volumes:
    - ./garage.toml:/etc/garage.toml:ro
    - outline_storage_data:/var/lib/garage/data
    - outline_storage_meta:/var/lib/garage/meta

volumes:
  outline_storage_data:
  outline_storage_meta:
```

```toml
# garage.toml (minimal single-node config)
metadata_dir = "/var/lib/garage/meta"
data_dir = "/var/lib/garage/data"
db_engine = "sqlite"
replication_factor = 1

[s3_api]
s3_region = "us-east-1"
api_bind_addr = "0.0.0.0:3900"

[admin]
api_bind_addr = "0.0.0.0:3903"
```

After Garage starts, create a key and bucket:

```bash
docker exec outline-storage garage key create outline-key
docker exec outline-storage garage bucket create outline
docker exec outline-storage garage bucket allow --read --write --key outline-key outline
```

Set in `outline.env`:
```bash
FILE_STORAGE=s3
AWS_ACCESS_KEY_ID=<garage-key-id>
AWS_SECRET_ACCESS_KEY=<garage-secret>
AWS_REGION=us-east-1
AWS_S3_UPLOAD_BUCKET_NAME=outline
AWS_S3_UPLOAD_BUCKET_URL=http://outline-storage:3900
AWS_S3_FORCE_PATH_STYLE=true
```

**Alternative: use `FILE_STORAGE=local`** — added in recent Outline versions. Attachments are stored on the container filesystem (mount a volume). Simpler for small teams; no separate S3 service required. Trade-off: no built-in CDN, no presigned URL delivery (files served through Outline itself), harder to scale horizontally.

## 6. Multi-Project Collections — Access Control

**Collections are the right abstraction for multi-project use.** Each project (KPI Light, future projects) gets its own collection. Within a collection, documents can be nested arbitrarily.

### Access control granularity

- **Collection-level permissions:** Read, Edit, or Admin — assignable per user or per group.
- **Groups:** Create a group per project team; add users; grant group access to specific collections. Recommended for teams > 10 users and for clear project separation.
- **Default access:** Each collection has a default access setting for all workspace members (public within workspace, or private). Private collections are invisible to non-members.
- **Document-level permissions:** As of recent versions, individual documents within a collection can also have restricted sharing. However, this is an additive layer; collection-level permissions are the primary control.

### Multi-project setup recommendation

```
Workspace
├── Collection: KPI Light            (access: kpi-light-team group)
│   ├── Dev Setup
│   ├── User Guide
│   ├── API Reference
│   └── Personio Runbook
├── Collection: Project B            (access: project-b-team group)
└── Collection: All-Hands            (access: all-workspace-members)
```

One workspace per company, multiple collections per project. This is the intended use case and works well at small team scale.

## 7. SMTP — Required or Optional?

**SMTP is optional for core functionality.** Outline runs without SMTP configured. With OIDC handling user provisioning, users can log in, create and edit documents, and collaborate without any email being sent.

**What SMTP enables:**
- Invitation emails (invite a new team member via email link)
- Document-change notification emails
- Digest emails

**What works without SMTP:**
- OIDC authentication and user creation (JIT provisioning on first login)
- All document editing, collaboration, and search

**Recommendation for v1.11 initial deploy:** Skip SMTP. Add it later if the team wants email notifications. Users join by authenticating via Dex — no invitation email needed.

**Hard requirement:** Even without SMTP, the OIDC provider **must** return an email claim. Outline uses email as a unique user identifier internally. If Dex federates from an LDAP source without email attributes, map a synthetic email (e.g., `uid@internal.example.com`) in Dex's connector claim mappings.

## 8. Markdown Import / Docs Seeding Workflow

Three options in increasing automation level:

### Option A: GUI drag-and-drop (simplest)

1. In Outline, open a collection.
2. Click "New Document" or drag `.md` files directly into the collection sidebar.
3. Outline converts Markdown to its prosemirror format on import.
4. Supports nested folder imports via `.zip` containing Markdown files.
5. Limitations: manual process, no CI integration.

### Option B: Outline REST API (programmatic, no extra tool)

Outline exposes a documented REST API at `https://wiki.internal/api/`. Key endpoints:

```bash
# Create a document
curl -X POST https://wiki.internal/api/documents.create \
  -H "Authorization: Bearer ${OUTLINE_API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Dev Setup",
    "collectionId": "<collection-uuid>",
    "text": "# Dev Setup\n\n...",
    "publish": true
  }'
```

Generate an API token in Outline Settings → API Tokens. Use `collectionId` from the collection URL.

For seeding from files:

```bash
for f in docs/*.md; do
  title=$(head -1 "$f" | sed 's/^# //')
  body=$(cat "$f")
  curl -s -X POST https://wiki.internal/api/documents.create \
    -H "Authorization: Bearer ${OUTLINE_API_TOKEN}" \
    -H "Content-Type: application/json" \
    -d "{\"title\": \"$title\", \"collectionId\": \"${COLLECTION_ID}\", \"text\": $(jq -Rs . < "$f"), \"publish\": true}"
done
```

### Option C: getoutline-cli (CI/CD pipeline)

`getoutline-cli` (PyPI) is an unofficial CLI designed for publishing Markdown files to Outline in CI pipelines.

```bash
pip install getoutline-cli

# .outline-cli.yml
outline_url: https://wiki.internal
api_token: ${OUTLINE_API_TOKEN}
collection_id: <collection-uuid>
documents:
  - path: docs/dev-setup.md
    title: Dev Setup
  - path: README.md
    title: KPI Light Overview
```

```bash
getoutline-cli publish
```

Supports regex substitutions (e.g., replace relative links with absolute wiki links before publishing).

### Import format notes

- Outline uses its own Markdown flavor; standard Markdown imports cleanly.
- Frontmatter (`---` YAML blocks) is stripped on import.
- Mermaid diagrams in fenced code blocks (`\`\`\`mermaid`) render natively.
- GitHub-style task lists (`- [ ] item`) are supported.
- Images referenced by URL render inline; images referenced by relative path need to be uploaded separately via the attachments API or replaced with absolute URLs.
